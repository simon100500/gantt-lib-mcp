# PRD: Fact GetGantt MVP

## Название

`fact.getgantt.ru` — отдельное мини-приложение для ежедневного сбора факта с площадки.

## Контекст

Текущий GetGantt хорошо подходит для руководителя проекта, планировщика и офиса, но перегружен для прораба. Прорабу не нужен полный график, зависимости, ресурсы, базовые планы и сложная аналитика. Ему нужен быстрый сценарий: открыть с телефона, увидеть работы на сегодня, отметить что сделано, что не сделано, что мешает, и отправить факт.

Цель продукта — сделать график живым: чтобы план регулярно подтверждался или опровергался фактом с площадки, а данные затем использовались в основном GetGantt для план-факта, статусов задач и последующего развития в сторону ресурсов и финансов.

## Проблема

Сегодня факт выполнения работ либо не собирается системно, либо собирается вне графика: в чатах, голосом, по фото, в Excel или в голове РП. Из-за этого:

- график быстро теряет связь с реальностью;
- РП не видит честную картину по отставаниям;
- прогресс задач не обновляется регулярно;
- причины отклонений не накапливаются в структурированном виде;
- в будущем сложно строить аналитику по ресурсам, деньгам и прогнозу.

## Цель MVP

Сделать максимально простой и быстрый сценарий сбора факта по секретной ссылке без логина:

1. РП создаёт ссылку на сбор факта.
2. Прораб открывает её на телефоне.
3. Видит задачи на сегодня.
4. По каждой задаче отмечает факт.
5. Отправляет.
6. Основной GetGantt автоматически получает обновлённый статус и прогресс задачи.

## Ключевая продуктовая гипотеза

Если дать прорабу отдельный лёгкий интерфейс без полного Ганта, то он будет регулярно вносить факт, а основной график начнёт отражать реальное состояние проекта.

## Целевая аудитория

### Основные пользователи

- прораб;
- мастер;
- бригадир;
- подрядчик, отвечающий за конкретный объём работ.

### Вторичные пользователи

- руководитель проекта;
- планировщик;
- офис и управление проектом.

## Позиционирование

- `GetGantt` — место, где планируют.
- `Fact GetGantt` — место, где подтверждают реальность.

## Сценарий MVP

### Сценарий для РП

1. Открывает проект в основном GetGantt.
2. Нажимает “Сбор факта”.
3. Создаёт секретную ссылку на работы на сегодня.
4. Копирует ссылку и отправляет её прорабу в мессенджер.

### Сценарий для прораба

1. Открывает ссылку на телефоне.
2. Вводит имя.
3. Видит список задач на сегодня.
4. Для каждой задачи отмечает:
   - не работали;
   - работали;
   - завершили;
   - проблема.
5. При необходимости указывает объём, причину и комментарий.
6. Отправляет факт.
7. Видит журнал отправленных записей.

### Сценарий для РП после отправки

1. Возвращается в основной GetGantt.
2. Видит обновлённый прогресс задач.
3. Видит дату последнего факта.
4. Может открыть историю факта по задаче.

## Обязательный функционал MVP

### 1. Отдельный домен

`fact.getgantt.ru`

Отдельный frontend с мобильным интерфейсом, работающий на текущем backend и общей БД.

### 2. Доступ по секретной ссылке

Без логина и без приглашения в проект.

Ссылка должна:

- открывать только fact-интерфейс;
- не давать доступ к полному проекту;
- не позволять менять план;
- не позволять видеть деньги, зависимости и структуру управления проектом.

### 3. Экран “Сегодня”

Главный экран MVP.

Показывает задачи, которые планово идут сегодня.

Для каждой карточки отображать:

- название задачи;
- родительский путь или группа;
- плановые даты;
- плановый объём и единицу, если есть;
- текущий прогресс.

### 4. Ввод факта по задаче

Для каждой задачи доступны статусы:

- работали;
- не работали;
- завершили;
- проблема.

### 5. Ввод объёма

Если у задачи есть единица измерения:

- ввод “сделано сегодня” числом.

Если единицы нет:

- выбор шага прогресса:
  - 0%;
  - 25%;
  - 50%;
  - 75%;
  - 100%.

### 6. Причина отклонения

Если задача не выполнялась или есть проблема, нужно дать выбор причины:

- нет материала;
- нет людей;
- не готов фронт;
- ждём смежников;
- изменение проектных решений;
- погода;
- техника;
- другое.

### 7. Комментарий

Свободный текст к факту по задаче.

### 8. Имя отправителя

При первом открытии ссылки пользователь вводит имя. Все записи по этой сессии отправляются с этим именем.

### 9. Журнал

Отдельный экран или вкладка со списком уже отправленных записей по этой ссылке.

Минимально показывать:

- дата;
- задача;
- статус;
- объём или прогресс;
- причина;
- комментарий.

### 10. Автоприменение факта

После отправки факт автоматически применяется к задаче в основном графике:

- обновляется `TaskProgressEntry`;
- пересчитывается `completedVolume`;
- пересчитывается `progress`;
- при необходимости обновляется `status`.

При этом сам факт хранится отдельно как первичная запись с площадки.

## Что НЕ входит в MVP

- фото;
- офлайн-режим;
- вход по телефону или коду;
- приглашение пользователя в проект;
- локации;
- недельный режим;
- план на завтра;
- закрытие дня;
- табель;
- стройконтроль;
- чек-листы качества;
- акты;
- мессенджер;
- отдельные проблемы без привязки к задаче.

## Ограничения MVP

- только leaf-задачи;
- только секретная ссылка;
- только сценарий “работы на сегодня”;
- без файлов и фото;
- без ручного подтверждения факта РП;
- без редактирования плановых данных из fact-интерфейса.

## Основные сущности

### FactAccessLink

Ссылка для публичного доступа к сбору факта.

Поля:

- `id`
- `token`
- `projectId`
- `label`
- `scopeType`
- `scopeTaskIds`
- `createdByUserId`
- `assigneeLabel`
- `expiresAt`
- `revokedAt`
- `createdAt`

### TaskFactReport

Сырая запись факта с площадки.

Поля:

- `id`
- `projectId`
- `taskId`
- `factAccessLinkId`
- `reportDate`
- `reporterName`
- `status`
- `inputMode`
- `quantityDone`
- `progressDeltaPercent`
- `blockerReason`
- `comment`
- `appliedAt`
- `createdAt`

## Правила применения факта

- `worked` и `done` влияют на прогресс задачи;
- `not_worked` и `blocked` не меняют объём, но сохраняются в истории;
- факт и официальное состояние задачи не должны быть одной и той же сущностью;
- одна запись факта по задаче и дате в рамках ссылки должна обновляться, а не дублироваться.

## Что должно появиться в основном GetGantt

На уровне задачи:

- текущий прогресс;
- дата последнего факта;
- количество fact-записей;
- история факта;
- признак наличия проблем.

## Метрики успеха MVP

### Продуктовые

- доля созданных ссылок, по которым был отправлен хотя бы один факт;
- среднее число задач, закрытых через fact-интерфейс;
- доля задач на сегодня, по которым внесён факт;
- среднее время отправки факта по ссылке.

### Качественные

- прораб может внести факт без объяснения интерфейса;
- РП видит обновление графика без ручного переноса данных из чатов;
- пользователи не путают fact-интерфейс с полным GetGantt.

## Критерии приёмки

1. РП может создать секретную ссылку на сбор факта по проекту.
2. Прораб может открыть ссылку без логина.
3. Прораб видит задачи на сегодня.
4. Прораб может внести статус, объём или шаг прогресса, причину и комментарий.
5. Факт сохраняется как отдельная запись.
6. Прогресс задачи в основном GetGantt обновляется автоматически.
7. Отозванная ссылка перестаёт работать.
8. Viewer не может создавать fact-ссылки.
9. Пользователь по ссылке не имеет доступа к остальному проекту.

## Будущие этапы после MVP

- фото;
- план на завтра и неделю;
- закрытие дня;
- локации;
- проблемы без привязки к задаче;
- invite-режим вместо публичной ссылки;
- подтверждение факта РП;
- офлайн-режим;
- аналитика по причинам отклонений;
- связка с ресурсами и финансами.
