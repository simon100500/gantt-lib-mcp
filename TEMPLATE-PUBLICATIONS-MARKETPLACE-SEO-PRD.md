# PRD: Публичные шаблоны и блоки для GetGantt

## 1. Контекст

В GetGantt уже есть важная основа: пользователь умеет собирать график как полноценный проект и умеет сохранять не только целый график, но и его часть как повторно используемый шаблонный блок. Это даёт правильное продуктовое ядро, но пока не превращено в публичную библиотеку, SEO-каталог и marketplace.

Сейчас есть три отдельных запроса, которые на самом деле должны решаться одной общей системой:

- SEO-каталог на `getgantt.ru` с посадочными страницами шаблонов;
- публичная библиотека и marketplace на `ai.getgantt.ru`;
- автоматический internal workflow, где агент или skill по текстовому запросу сам генерирует шаблон, публикует его и создаёт страницу.

Если решать эти задачи разрозненно, быстро появится дублирование:

- один формат шаблонов для сайта;
- другой формат для web-продукта;
- ручная синхронизация между проектами, страницами и каталогом;
- отсутствие единого жизненного цикла шаблона.

Нужен единый продуктовый слой публикаций, поверх которого будут работать и сайт, и marketplace, и automation.

## 2. Проблема

Сегодня проект в GetGantt является живым редактируемым объектом. Это хорошо для работы пользователя, но плохо как единственный источник публичного шаблона.

Если напрямую использовать проект как публичный шаблон:

- любая незавершённая правка моментально меняет публичный результат;
- нельзя чётко разделить черновик и опубликованную версию;
- сложно делать модерацию;
- невозможно нормально строить marketplace с user-generated шаблонами;
- трудно поддерживать SEO-страницы, где нужен стабильный URL, стабильный контент и контролируемая выдача.

Кроме того, шаблон в системе бывает двух разных типов:

- целый проектный шаблон;
- частичный шаблонный блок, например только электрика, фундамент, фасад, отделка, запуск оборудования, ввод в эксплуатацию.

Обе сущности должны жить в одной общей модели, попадать в каталог, участвовать в marketplace и быть пригодными для вставки в пользовательский график.

## 3. Цель продукта

Построить единую систему публичных публикаций шаблонов, которая позволяет:

- публиковать как целые шаблоны проектов, так и частичные блоки;
- использовать одну и ту же базу публикаций для SEO-сайта и для marketplace в основном продукте;
- разделять редактируемый source-график и опубликованную версию;
- поддерживать модерацию, верификацию и editorial selection;
- поддерживать автоматическую генерацию и публикацию через internal skill;
- в будущем открыть публикацию шаблонов для обычных пользователей.

## 4. Ключевая продуктовая идея

Нужна отдельная сущность публикации шаблона.

### Базовая модель

- `Project` — рабочий редактируемый график, где автор собирает и улучшает структуру.
- `Template Publication` — опубликованная, зафиксированная публичная версия шаблона или блока.

Проект отвечает на вопрос:

- "что сейчас редактирует автор?"

Публикация отвечает на вопрос:

- "что сейчас доступно в каталоге, на сайте и в marketplace?"

Это принципиальное разделение.

## 5. Основные сущности

### 5.1 Project

Рабочий график в продукте.

Свойства:

- редактируется пользователем или агентом;
- может использоваться как source для публикации;
- не обязан быть публичным;
- может иметь много публикаций в разные моменты времени.

### 5.2 Template Publication

Публичный объект каталога.

Свойства:

- содержит snapshot задач и зависимостей на момент публикации;
- имеет собственный URL-идентификатор;
- содержит SEO-метаданные и marketplace-метаданные;
- имеет жизненный цикл публикации, модерации и архивирования;
- может происходить как из целого проекта, так и из выбранного поддерева задач.

### 5.3 Publication Kind

У каждой публикации должен быть тип:

- `template` — шаблон целого проекта или законченного сценария;
- `block` — вставляемый строительный или проектный модуль.

Примеры:

- `template`: строительство поликлиники, ремонт квартиры под ключ, строительство частного дома;
- `block`: электрика, устройство фундамента, фасадные работы, монтаж вентиляции, благоустройство.

### 5.4 Verification Layer

Публикация должна иметь независимый слой доверия:

- `unverified`;
- `reviewed`;
- `verified`;
- `editorial`.

Это позволит:

- показывать в marketplace все опубликованные и допустимые публикации;
- показывать на SEO-сайте только отобранные и проверенные.

## 6. Продуктовые сценарии

### 6.1 Internal editorial flow

Сценарий для команды GetGantt:

1. Создать или сгенерировать source-проект.
2. Доработать график вручную или через AI.
3. Опубликовать как `template` или `block`.
4. Указать SEO-поля и каталогизацию.
5. Отметить публикацию как `verified` или `editorial`.
6. Получить:
   - карточку в marketplace;
   - SEO-страницу на сайте;
   - возможность создавать пользовательские проекты из этой публикации.

### 6.2 User marketplace flow

Сценарий для обычного пользователя:

1. Пользователь создаёт проект или сохраняет часть графика как reusable source.
2. Нажимает "Опубликовать в marketplace".
3. Заполняет title, category, short description, tags и тип `template` или `block`.
4. Система создаёт публикацию со snapshot.
5. Публикация попадает в marketplace со статусом `published`, но остаётся `unverified`.
6. На SEO-сайт такая публикация не попадает, пока не пройдет review.

### 6.3 Consumer flow в marketplace

1. Пользователь открывает раздел шаблонов на `ai.getgantt.ru`.
2. Фильтрует по типу:
   - шаблоны;
   - блоки.
3. Фильтрует по категории, отрасли, сложности, длительности, роли.
4. Открывает карточку публикации.
5. Видит preview-график, структуру работ, пояснения, метки "verified" или "editorial".
6. Выбирает:
   - создать новый проект из шаблона;
   - вставить блок в текущий проект.

### 6.4 Consumer flow на SEO-сайте

1. Пользователь приходит из поиска на посадочную страницу.
2. Видит понятный заголовок, график, список работ, зависимости, рекомендации, FAQ.
3. Понимает, что это либо:
   - готовый шаблон проекта;
   - отдельный блок работ.
4. Переходит в `ai.getgantt.ru` и создаёт проект или использует блок в текущем графике.

### 6.5 Internal skill flow

Текстовый сценарий:

`добавь шаблон по строительству поликлиники`

Skill должен уметь:

1. Сгенерировать source-график через существующие product API.
2. Создать source-проект в системном контуре шаблонов.
3. Сохранить задачи, зависимости, календарь и связанные метаданные.
4. Сформировать publication-метаданные.
5. Создать `template_publication`.
6. При необходимости сразу пометить её `verified/editorial`.
7. Обновить каталог для SEO и marketplace.

## 7. Что считается шаблоном, а что блоком

Система не должна ограничиваться только полными проектами.

### Template

Используется, когда пользователь хочет получить целую стартовую структуру проекта.

Признаки:

- имеет собственный корневой сценарий;
- покрывает значимую часть жизненного цикла проекта;
- обычно используется для создания нового проекта.

### Block

Используется, когда пользователь хочет вставить в существующий график модуль работ.

Признаки:

- описывает часть проекта;
- может повторяться в разных шаблонах и проектах;
- обычно используется для вставки в текущий проект;
- может иметь собственную SEO-страницу и карточку в marketplace.

Важно: `block` не является второсортным объектом. Это такой же публичный reusable asset, как и `template`.

## 8. Требования к публикации

Каждая публикация должна хранить:

- стабильный `slug`;
- `kind`: `template` или `block`;
- `title`;
- `subtitle` или short summary;
- `category`;
- `industry` или тематический сегмент;
- `tags`;
- `sourceProjectId`;
- `sourceUserId`;
- `sourceSelection`, если публикация создана не из всего проекта, а из части;
- `publishedSnapshot` задач и зависимостей;
- `ganttDayMode`, calendar settings, timeline markers, если они важны для воспроизведения;
- редакционный текст;
- SEO title;
- SEO description;
- preview summary для marketplace;
- cover/meta для карточки;
- `status`;
- `visibility`;
- `verificationStatus`;
- `publishedAt`;
- `updatedAt`.

## 9. Предлагаемая модель статусов

### Publication Status

- `draft` — публикация создана, но ещё не видна пользователям;
- `published` — доступна в продукте;
- `archived` — снята с публикации;
- `rejected` — отклонена модерацией.

### Visibility

- `private` — видит только автор и админ;
- `marketplace` — видна в marketplace;
- `site` — разрешена для SEO-сайта;
- `both` — и marketplace, и SEO-сайт.

### Verification Status

- `unverified` — пользовательская или непроверенная;
- `reviewed` — просмотрена командой;
- `verified` — подтверждена как качественная;
- `editorial` — вручную отобрана как featured content.

## 10. Почему публикация должна быть snapshot, а не live project

Публичный каталог не должен напрямую зависеть от текущего состояния живого проекта.

Причины:

- автор может редактировать source в несколько приёмов;
- AI может временно ухудшить структуру;
- незавершённая правка не должна ломать публичную страницу;
- marketplace и SEO-сайт должны показывать предсказуемую версию;
- нужны review, rollback и version awareness.

Правильная модель:

- source-проект живёт как черновик;
- publication создаётся как зафиксированный snapshot;
- новая правка source-проекта требует републикации.

Это не бюрократия. Это базовое условие для качественной публичной библиотеки.

## 11. Структура каталога на сайте

На `getgantt.ru` нужен статический SEO-каталог, который использует публикации как источник правды.

### Страницы

- общий каталог шаблонов и блоков;
- категории;
- отдельные посадочные страницы;
- возможно, подборки по типу работ или отрасли.

### Для каждой страницы публикации

Нужно показывать:

- H1 и value proposition;
- график-preview;
- список этапов или работ;
- зависимости и логика последовательности;
- когда уместно использовать шаблон или блок;
- типичные ошибки;
- FAQ;
- CTA в продукт.

### Что попадает на сайт

На SEO-сайт попадают не все публикации, а только те, которые:

- имеют `status = published`;
- имеют `visibility = site` или `both`;
- имеют `verificationStatus = verified` или `editorial`.

## 12. Структура marketplace в продукте

На `ai.getgantt.ru` нужен полноценный раздел шаблонов.

### Раздел должен поддерживать

- единый каталог;
- переключение между `templates` и `blocks`;
- карточки публикаций;
- поиск;
- фильтры;
- сортировки;
- preview-график;
- действия:
  - создать новый проект;
  - вставить блок в текущий проект;
  - сохранить к себе;
  - у автора в будущем обновить публикацию.

### Отображение trust

Marketplace должен явно показывать:

- авторский шаблон;
- проверенный шаблон;
- редакционный шаблон.

Это важно и для доверия, и для монетизации в будущем.

## 13. Автоматизация через skill

Система должна быть спроектирована так, чтобы ею можно было управлять текстовыми командами.

### Пример целевого workflow

`добавь шаблон по строительству поликлиники`

Ожидаемое поведение:

1. Агент определяет, что нужен `template`.
2. Генерирует или собирает source-проект.
3. Создаёт source-проект в системном шаблонном контуре.
4. Генерирует publication snapshot.
5. Генерирует title, summary, tags, category, SEO title, SEO description, page body.
6. Создаёт publication.
7. Помечает visibility и verification policy по контексту.
8. Обновляет связанный каталог сайта и marketplace.

### Пример для блока

`добавь блок по электрике для жилого дома`

Ожидаемое поведение:

1. Агент определяет, что нужен `block`.
2. Генерирует только соответствующий подграф работ.
3. Публикует block с подходящими метаданными.
4. Block становится доступен и в SEO-каталоге, и в marketplace.

## 14. Контур владения шаблонами

На первом этапе нужен системный контур шаблонов.

Рекомендуемая модель:

- отдельный технический или системный пользователь для internal editorial template flow;
- отдельное рабочее пространство или группы source-проектов шаблонов;
- отдельные admin actions для публикации и верификации.

Этот системный пользователь нужен не как основная продуктовая сущность для marketplace, а как надёжный internal owner для:

- автоматической генерации;
- редакционной библиотеки;
- управляемых шаблонов команды GetGantt.

При этом user-generated публикации должны поддерживаться с самого начала на уровне модели публикаций.

## 15. API и системные контракты

Нужны как минимум четыре группы контрактов.

### 15.1 Source management

- создать source-проект для шаблона;
- сохранить selection как source для блока;
- получить source snapshot.

### 15.2 Publication management

- создать publication из проекта или selection;
- обновить publication metadata;
- republish publication из обновлённого source;
- архивировать publication;
- сменить verification и visibility.

### 15.3 Public catalog API

- список публикаций;
- детали публикации по `slug`;
- filtered listing для marketplace;
- filtered listing для SEO-build pipeline.

### 15.4 Consumption API

- создать новый проект из publication;
- вставить block-publication в текущий проект;
- получить preview snapshot публикации.

## 16. Метрики успеха

### Product metrics

- количество опубликованных `template` публикаций;
- количество опубликованных `block` публикаций;
- конверсия из SEO-страницы в открытие продукта;
- конверсия из карточки marketplace в создание проекта;
- конверсия из block-card в вставку блока;
- доля verified публикаций;
- повторное использование блоков в пользовательских проектах.

### Operational metrics

- время от команды skill до готовой публикации;
- процент публикаций, требующих ручной правки;
- количество отклонённых публикаций;
- время модерации публикации.

## 17. In Scope

- единая сущность `template publication`;
- поддержка `template` и `block`;
- snapshot-публикация из проекта или поддерева задач;
- marketplace на `ai.getgantt.ru`;
- SEO-каталог на `getgantt.ru`;
- visibility, verification и moderation layer;
- internal skill flow для автогенерации и публикации;
- user-generated публикации как часть общей модели.

## 18. Out of Scope для первой полной версии

- платные шаблоны и revenue share;
- рейтинги, отзывы и social proof от пользователей;
- version diff UI между публикациями;
- автоматическая AB-оптимизация SEO-страниц;
- полноценный open creator economy.

Эти возможности должны быть совместимы с моделью, но не обязательны для первого релиза.

## 19. Основные продуктовые решения

### Решение 1

Публичный шаблон не равен проекту.

Публичный шаблон — это отдельная публикация, которая создаётся из проекта.

### Решение 2

Система изначально поддерживает два вида публикаций:

- `template`;
- `block`.

### Решение 3

Сайт и marketplace не имеют отдельных независимых источников данных.

Они используют одну и ту же базу публикаций с разными правилами отбора.

### Решение 4

На SEO-сайт попадают только проверенные публикации.

Marketplace может показывать более широкий слой опубликованных работ.

### Решение 5

Automation и skill-flow не являются внешним костылём.

Они должны опираться на те же source и publication API, что и UI.

## 20. Критерии готовности

Система считается концептуально готовой, когда:

1. можно создать publication из полного проекта;
2. можно создать publication из части проекта;
3. publication имеет тип `template` или `block`;
4. publication можно показать в marketplace;
5. publication можно показать на SEO-сайте;
6. publication можно использовать для создания нового проекта;
7. block-publication можно вставить в текущий проект;
8. публикация может быть verified/editorial;
9. internal skill может создать и опубликовать новую сущность без ручного редактирования кода.

## 21. Следующий шаг после PRD

После этого PRD следующей рабочей задачей должен стать технический дизайн:

- Prisma schema для `template_publications`;
- source-to-publication pipeline;
- public API contracts;
- admin and skill publish flow;
- site build integration;
- marketplace UX contract.
