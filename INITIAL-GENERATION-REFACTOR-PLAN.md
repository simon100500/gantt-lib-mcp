# Рефакторинг AI-генерации стартового графика (v2)

## Кратко
Нужно вернуть initial generation к AI-first подходу и убрать текущий deterministic fast path для широких запросов в пустом проекте. Целевой режим: при первом запросе на построение графика система использует **двухэтапный pipeline (planning → execution)** с разными моделями, подмешивает в planning-этап доменный контекст из серверного справочника, оценивает качество плана между этапами и принимает только такой график, который даёт предметный "вау-эффект", а не просто валидную мутацию. Уточняющих вопросов пользователю не задаём — даже расплывчатый запрос обрабатываем молча, опираясь на собственную доменную экспертизу.

## Ключевые изменения

### Убрать deterministic fast path
- Убрать из серверного пайплайна перехват broad-запросов через `initial_schedule_template`.
- Удалить текущую логику, которая по широкому запросу в пустом проекте сразу создаёт generic construction template.
- Не заменять её новыми regexp-ветками для контентной генерации.
- Быстрые deterministic path допустимы позже только для узких, явно безопасных кейсов, где они не подменяют продуктовый AI-опыт.

### Ввести отдельный режим `initial_generation`
- Условие входа: проект пустой и запрос относится к широкому построению графика.
- Этот режим должен отличаться от обычного mutation flow не только intent-классификацией, но и orchestration-логикой, prompt strategy, quality gate и model routing.

### ⟁ Двухэтапный pipeline: Planning → Execution

Заменить текущий одношаговый подход (один LLM-вызов генерирует мутации) на два последовательных этапа с разными ролями, моделями и форматами.

**Этап 1 — Planning (сильная модель)**

- Вход: запрос пользователя + server brief (см. ниже) + контекст из доменного справочника.
- Задача: сгенерировать **структурный план графика** — не мутации, а outline проекта.
- Выход — structured JSON/object:
  - фазы проекта (с предметными названиями)
  - задачи внутри фаз (иерархия)
  - примерные длительности (в рабочих днях)
  - логика последовательности и зависимостей между фазами
  - где уместно — параллельные потоки работ
- Модель: сильная (например, claude-sonnet-4 / opus-class). Это оправдано, потому что output компактный (план, а не мутации), а от качества этого шага зависит весь результат.
- Latency budget: ~5–8 секунд из общих 10–20.

**Quality gate между этапами**

- Оценка плана по критериям (можно rule-based, без дополнительного LLM-вызова):
  - количество фаз ≥ N (зависит от типа объекта)
  - есть иерархия (не плоский список)
  - названия специфичны (не "Этап 1", "Работа 2")
  - есть логичная последовательность (не всё параллельно и не всё последовательно)
  - длительности в разумном диапазоне для типа объекта
- Если план слабый → repair на уровне плана (передать модели план + конкретную критику, попросить переделать). Repair плана дешевле и эффективнее, чем repair готовых мутаций.
- Ограничение: 1 repair-цикл на уровне плана.

**Этап 2 — Execution (дешёвая модель)**

- Вход: утверждённый план из этапа 1 + текущее состояние проекта (пустой) + mutation tools.
- Задача: **механически** превратить план в конкретные мутации — создать задачи, расставить даты, длительности, зависимости.
- Модель: дешёвая / быстрая (например, haiku-class). Задача формализована планом, креативность не нужна.
- Latency budget: ~3–5 секунд.
- Если мутации невалидны технически (ошибки ID, невалидные даты), допустим 1 технический retry.

### Серверный AI-brief и доменный справочник

**Brief перед planning-этапом:**
- Brief должен быть коротким и структурным, не в виде готового шаблона задач.
- В brief включать:
  - предполагаемый тип объекта из запроса
  - масштабные сигналы из текста пользователя
  - ожидание на предметность иерархии
  - требование создать не фрагмент, а стартовый каркас проекта
  - требование избегать generic filler вроде `Работа 1/2/3`
- Brief не должен жёстко навязывать список работ; он должен направлять модель, а не заменять её.

**⟁ Доменный справочник (knowledge injection):**
- Для каждого распознанного типа объекта (жилой дом, детский сад, офисное здание, ремонт и т.д.) хранить компактный серверный reference:
  - типичные фазы строительства для этого типа объекта
  - характерные особенности (например, для детского сада — согласования с надзорными органами; для частного дома — инженерные сети до закрытия контура)
  - ожидаемый масштаб (порядок количества задач, диапазон длительности проекта)
  - типичные параллельные потоки
- Справочник подмешивается в prompt planning-этапа как context, а не как жёсткий шаблон.
- Если тип объекта не распознан (расплывчатый запрос вроде "построй график") — использовать generic construction reference с пометкой "адаптируй под наиболее вероятную интерпретацию".
- Справочник НЕ является curated template. Он не содержит готовых списков задач — только доменные ориентиры.

### Маршрутизация модели
- Этап 1 (planning) → сильная модель.
- Этап 2 (execution) → дешёвая модель.
- Обычные последующие правки (не initial generation) → дешёвая модель как сейчас.
- Routing опирается на класс запроса и состояние проекта, а не на список фраз.

### ⟁ Без уточняющих вопросов
- При расплывчатом запросе в пустом проекте система НЕ задаёт уточняющих вопросов.
- Логика: расплывчатый запрос, скорее всего, означает, что пользователь тестирует сервис. Конкретный пользователь напишет конкретный запрос.
- Вместо вопросов — опора на доменную экспертизу: server brief + справочник + сильная модель на planning-этапе.
- Система должна выдать максимально предметный результат даже по запросу "построй график строительства".

### Repair loop (пересмотрен под двухэтапность)
- **На уровне плана (между этапами 1 и 2):** если quality gate не прошёл, repair prompt с конкретной критикой:
  - недостаточная предметность под тип объекта
  - слишком маленький охват
  - слабая иерархия
  - generic naming
  - нереалистичная последовательность
- Максимум 1 repair-цикл на этапе плана.
- **На уровне мутаций (после этапа 2):** только технический repair — невалидные ID, ошибки формата, сломанные зависимости. Продуктовый repair на этом уровне не нужен, если план был хороший.
- Максимум 1 технический retry на этапе мутаций.
- Общий бюджет: 2–4 LLM-вызова в худшем случае (plan + plan repair + execute + execute retry), целевой latency 10–20 секунд.

### Prompt contract
- Явно разделить prompt'ы для двух этапов:
  - **Planning prompt:** ты — эксперт по строительному планированию. На основе запроса и справочника создай структурный план проекта. Требования: предметные этапы, чистая иерархия, реалистичная последовательность, отсутствие placeholder naming.
  - **Execution prompt:** вот утверждённый план проекта. Создай мутации для его реализации. Все задачи, даты, длительности и зависимости должны точно следовать плану.
- Для planning не использовать fallback-инструкции, которые толкают модель к safe minimal output.

### Логирование
- Логировать:
  - какой route выбран (initial_generation vs mutation)
  - этап 1: модель, object type, сгенерированный план, verdict quality gate, repair reason (если был)
  - этап 2: модель, количество мутаций, технические ошибки, retry reason (если был)
  - финальный результат: принят / отклонён, причина
- По одному run должно быть видно: запрос → тип объекта → план → критика (если была) → мутации → результат.

## Интерфейсы и внутренняя структура
- Добавить явный внутренний тип/флаг режима генерации: `initial_generation`.
- Добавить типизированные структуры для:
  - `GenerationBrief` — brief + доменный контекст из справочника
  - `ProjectPlan` — структурированный output этапа 1 (фазы, задачи, иерархия, зависимости)
  - `PlanQualityVerdict` — результат quality gate между этапами
  - `RepairReason` — почему план отправлен на доработку
  - `ModelRoutingDecision` — какая модель выбрана и почему
- Не добавлять новые пользовательские MCP-tools на этом этапе.
- Вся реализация остаётся поверх текущих нормализованных mutation tools.

## Проверки

### Тесты маршрутизации
- Пустой проект + broad generation request → `initial_generation` с двухэтапным pipeline.
- `Построй график строительства детского садика` → НЕ уходит в deterministic template path.
- Обычные edit-запросы → НЕ получают двухэтапный routing.
- Расплывчатый запрос → НЕ порождает уточняющих вопросов, а идёт в генерацию.

### Тесты planning-этапа
- Object type корректно попадает в server brief.
- Доменный справочник подмешивается в prompt для распознанного типа объекта.
- Для нераспознанного типа — подмешивается generic construction reference.
- Output planning-этапа — валидный structured format (JSON/object с фазами, задачами, зависимостями).

### Тесты quality gate (между этапами)
- План без иерархии → repair.
- План с generic naming → repair.
- Предметный план с нормальной иерархией → принят, идёт на execution.
- После 1 repair-цикла план всё ещё слабый → принять лучший вариант, не зацикливаться.

### Тесты execution-этапа
- Утверждённый план → корректные мутации (задачи, даты, зависимости соответствуют плану).
- Невалидные мутации → 1 технический retry.
- После retry мутации всё ещё невалидны → вернуть ошибку, не молчать.

### Тесты логов
- В логах видны: route, model tier для обоих этапов, план, quality verdict, repair reason, acceptance reason.
- По одному run можно восстановить полный путь генерации.

### Ручные сценарии
- `Построй типичный график строительства`
- `Построй график строительства детского садика`
- `Построй график ремонта офиса 300 м2`
- `Построй график строительства частного дома из газобетона`
- `Построй график` (минимальный запрос — система справляется без уточнений)
- Для каждого сравнивать: предметность, иерархию, реалистичность последовательности, соответствие доменному справочнику и итоговый "вау-эффект".

## Допущения
- Для initial generation допустим latency 10–20 секунд (2–4 LLM-вызова суммарно).
- Planning-этап идёт на сильной модели, execution — на дешёвой.
- Система не задаёт уточняющих вопросов; расплывчатый запрос обрабатывается с максимальной опорой на доменную экспертизу.
- После слабого плана — 1 внутренний repair, а не запрос к пользователю.
- Текущие deterministic-изменения с generic construction template откатываются.
- Доменный справочник — это NOT curated templates. Это compact reference для направления модели, а не готовые списки задач.
- Curated templates — отдельная фаза, не смешивать с AI-first архитектурой.
