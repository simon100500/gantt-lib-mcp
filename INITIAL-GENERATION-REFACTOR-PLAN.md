# Рефакторинг AI-генерации стартового графика (v2)

## Кратко
Нужно вернуть initial generation к AI-first подходу и убрать текущий deterministic fast path для широких запросов в пустом проекте. Целевой режим: при первом запросе на построение графика система использует **двухэтапный pipeline (planning → deterministic execution)** с разными моделями, подмешивает в planning-этап доменный контекст из серверного справочника, оценивает качество плана между этапами и принимает только такой график, который даёт предметный "вау-эффект", а не просто валидную мутацию. Уточняющих вопросов пользователю не задаём — даже расплывчатый запрос обрабатываем молча, опираясь на собственную доменную экспертизу. Для `v1` scope ограничен прежде всего стройкой и ремонтом; multi-domain initial generation не является целью этого рефакторинга.

## Ключевые изменения

### Убрать deterministic fast path
- Убрать из серверного пайплайна перехват broad-запросов через `initial_schedule_template`.
- Удалить текущую логику, которая по широкому запросу в пустом проекте сразу создаёт generic construction template.
- Не заменять её новыми regexp-ветками для контентной генерации.
- Быстрые deterministic path допустимы позже только для узких, явно безопасных кейсов, где они не подменяют продуктовый AI-опыт.

### Ввести отдельный режим `initial_generation`
- Условие входа: проект пустой и запрос относится к широкому построению графика.
- Этот режим должен отличаться от обычного mutation flow не только intent-классификацией, но и orchestration-логикой, prompt strategy, quality gate и model routing.

### ⟁ Двухэтапный pipeline: Planning → Deterministic Execution

Заменить текущий одношаговый подход (один LLM-вызов генерирует мутации) на два последовательных этапа с разными ролями и форматами. Первый этап остаётся AI-центричным, второй исполняется на сервере без MCP-агента.

**Этап 1 — Planning (сильная модель)**

- Вход: запрос пользователя + server brief (см. ниже) + контекст из доменного справочника.
- Задача: сгенерировать **структурный план графика** — не мутации, а outline проекта.
- Выход — structured JSON/object (`ProjectPlan`), пригодный для алгоритмического исполнения на сервере:
  - устойчивые ключи узлов (`nodeKey`) для ссылок между задачами
  - фазы проекта (с предметными названиями)
  - задачи внутри фаз (иерархия через `parentNodeKey`)
  - примерные длительности (в рабочих днях)
  - логика последовательности и зависимостей между узлами
  - где уместно — параллельные потоки работ
- Модель: сильная (например, claude-sonnet-4 / opus-class). Это оправдано, потому что output компактный (план, а не мутации), а от качества этого шага зависит весь результат.
- Latency budget: ~5–8 секунд из общих 10–20.

**Quality gate между этапами**

- Оценка плана по критериям (можно rule-based, без дополнительного LLM-вызова):
  - количество фаз ≥ N (зависит от типа объекта)
  - есть иерархия (не плоский список)
  - названия специфичны (не "Этап 1", "Работа 2")
  - есть логичная последовательность (не всё параллельно и не всё последовательно)
  - длительности в разумном диапазоне для типа объекта
- Если план слабый → repair на уровне плана (передать модели план + конкретную критику, попросить переделать). Repair плана дешевле и эффективнее, чем repair готовых мутаций.
- Ограничение: 1 repair-цикл на уровне плана.

**Этап 2 — Deterministic Execution (серверный compiler)**

- Вход: утверждённый `ProjectPlan` из этапа 1 + текущее состояние проекта (пустой) + серверные правила компиляции.
- Задача: **механически** превратить план в `create_tasks_batch` и выполнить его напрямую через `commandService`, в обход MCP-агента.
- Сервер:
  - валидирует `ProjectPlan` по schema
  - генерирует реальные task IDs для каждого `nodeKey`
  - вычисляет даты и зависимости по правилам компиляции
  - собирает один authoritative batch command
  - коммитит его через `commandService`
- Правила компиляции для `v1`:
  - старт расписания берётся от текущей даты на сервере;
  - по умолчанию используется режим рабочих дней;
  - по умолчанию используется уже существующий производственный календарь РФ;
  - `durationDays` интерпретируется в рабочих днях;
  - дочерние задачи получают реальные даты, а parent phase не планируется отдельно и получает диапазон только rollup'ом от детей;
  - поддерживаются все типы зависимостей `FS/SS/FF/SF`;
  - planning-модель должна в основном использовать `FS`, но compiler обязан уметь исполнить и остальные типы, если они пришли в валидном плане.
- Execution не должен зависеть от второго LLM-вызова в happy path.
- Если план не удаётся скомпилировать целиком технически (битые ссылки, цикл зависимостей, невозможная схема) — compiler должен сначала пытаться собрать максимально полный **partial schedule**, а не сразу завершаться пустой ошибкой.
- Если удаётся безопасно отбросить только проблемные узлы/связи и сохранить предметный каркас проекта, partial schedule коммитится.
- Если после отбрасывания проблемных частей результат становится слишком бедным, partial schedule не коммитится и система возвращает controlled error.
- Нельзя молча переключаться в MCP-agent mutation flow как fallback.
- Partial schedule считается допустимым только если одновременно выполняются условия:
  - сохранилось не меньше 60% исходных узлов плана;
  - сохранилось не меньше 3 верхнеуровневых фаз;
  - у каждой сохранённой верхнеуровневой фазы есть хотя бы один дочерний task;
  - после очистки нет битых зависимостей, ссылок на удалённые узлы и пустых контейнеров.
- Если partial schedule закоммичен, финальный ответ пользователю должен честно сообщать, что стартовый график был построен частично, но без технических деталей про compiler internals.

### Серверный AI-brief и доменный справочник

**Brief перед planning-этапом:**
- Brief должен быть коротким и структурным, не в виде готового шаблона задач.
- В brief включать:
  - предполагаемый тип объекта из запроса
  - масштабные сигналы из текста пользователя
  - ожидание на предметность иерархии
  - требование создать не фрагмент, а стартовый каркас проекта
  - требование избегать generic filler вроде `Работа 1/2/3`
- Brief не должен жёстко навязывать список работ; он должен направлять модель, а не заменять её.

**⟁ Доменный справочник (knowledge injection):**
- Для каждого распознанного типа объекта (жилой дом, детский сад, офисное здание, ремонт и т.д.) хранить компактный серверный reference:
  - типичные фазы строительства для этого типа объекта
  - характерные особенности (например, для детского сада — согласования с надзорными органами; для частного дома — инженерные сети до закрытия контура)
  - ожидаемый масштаб (порядок количества задач, диапазон длительности проекта)
  - типичные параллельные потоки
- Справочник подмешивается в prompt planning-этапа как context, а не как жёсткий шаблон.
- Если тип объекта не распознан (расплывчатый запрос вроде "построй график") — использовать generic construction reference с дефолтной интерпретацией **"частный жилой дом"** как наиболее вероятным и продуктово сильным baseline для `v1`.
- В planning prompt это указывать явно: если уверенный тип объекта не найден, интерпретируй запрос как типичный проект строительства частного жилого дома, но не выдумывай узкоспециализированные работы без опоры.
- Справочник НЕ является curated template. Он не содержит готовых списков задач — только доменные ориентиры.

### Маршрутизация модели
- `initial_generation` planning → сильная модель.
- Обычные последующие правки (не initial generation) → дешёвая модель по умолчанию.
- Routing опирается на класс запроса и состояние проекта, а не на список фраз.
- Добавить явную server-side конфигурацию маршрутизации моделей:
  - `OPENAI_MODEL` / `model` — основная модель; используется как strong model для initial planning;
  - `OPENAI_CHEAP_MODEL` / `cheap_model` — опциональная дешёвая модель для обычных правок и mutation-agent flow;
  - API endpoint и ключ остаются общими.
- Не пытаться менять модель "внутри" уже запущенной agent-сессии. Если SDK фактически фиксирует модель на один `query()`-run, routing должен происходить **до** старта конкретного вызова.
- Следствие для архитектуры: initial planning и обычный mutation-agent должны вызываться как отдельные runs с явным выбором модели на старте.
- Если `cheap_model` не задана, весь routing деградирует в single-model mode без падения сервиса: для mutation-agent используется основная `model`.
- Routing decision принимается в точке старта конкретного SDK/tool run и логируется как отдельный `ModelRoutingDecision`.

### ⟁ Без уточняющих вопросов
- При расплывчатом запросе в пустом проекте система НЕ задаёт уточняющих вопросов.
- Логика: расплывчатый запрос, скорее всего, означает, что пользователь тестирует сервис. Конкретный пользователь напишет конкретный запрос.
- Вместо вопросов — опора на доменную экспертизу: server brief + справочник + сильная модель на planning-этапе.
- Система должна выдать максимально предметный результат даже по запросу "построй график строительства".

### Repair loop (пересмотрен под двухэтапность)
- **На уровне плана (между этапами 1 и 2):** если quality gate не прошёл, repair prompt с конкретной критикой:
  - недостаточная предметность под тип объекта
  - слишком маленький охват
  - слабая иерархия
  - generic naming
  - нереалистичная последовательность
- Максимум 1 repair-цикл на этапе плана.
- **На уровне execution compiler:** только техническая валидация и compile-time ошибки — битые ссылки, цикл в зависимостях, невалидные даты, невозможная иерархия.
- На этом уровне не делать продуктовый repair через mutation-agent. Продуктовое качество должно решаться на уровне плана.
- Отдельный LLM-repair по compile-ошибкам в `v1` не добавлять. Иначе двухэтапный pipeline снова превращается в скрытый многоэтапный orchestration.
- Общий бюджет: 1–2 LLM-вызова в happy path / repair path (plan + optional plan repair), целевой latency 10–20 секунд.

### Prompt contract
- Явно разделить prompt'ы для двух этапов:
  - **Planning prompt:** ты — эксперт по строительному планированию. На основе запроса и справочника создай структурный план проекта в строгом machine-readable формате. Требования: предметные этапы, чистая иерархия, реалистичная последовательность, отсутствие placeholder naming.
  - **Repair prompt:** вот предыдущий план и конкретная критика quality gate. Верни полностью исправленный `ProjectPlan`, а не комментарии.
- Для planning не использовать fallback-инструкции, которые толкают модель к safe minimal output.

### Логирование
- Логировать:
  - какой route выбран (initial_generation vs mutation)
  - этап 1: модель, object type, сгенерированный план, verdict quality gate, repair reason (если был)
  - этап 2: compile verdict, размер batch command, количество задач, технические ошибки компиляции (если были)
  - финальный результат: принят / отклонён, причина
- По одному run должно быть видно: запрос → тип объекта → план → критика (если была) → compiled command → результат.

## Интерфейсы и внутренняя структура
- Добавить явный внутренний тип/флаг режима генерации: `initial_generation`.
- Добавить типизированные структуры для:
  - `GenerationBrief` — brief + доменный контекст из справочника
  - `ProjectPlan` — структурированный output этапа 1 (фазы, задачи, иерархия, зависимости, scheduling metadata)
  - `PlanQualityVerdict` — результат quality gate между этапами
  - `RepairReason` — почему план отправлен на доработку
  - `ModelRoutingDecision` — какая модель выбрана и почему
  - `CompiledInitialSchedule` — результат компиляции `ProjectPlan` в server-side create batch
- Не добавлять новые пользовательские MCP-tools на этом этапе.
- `initial_generation` исполняется не поверх текущих нормализованных mutation tools, а напрямую через `commandService`.
- Обычный mutation flow через MCP-agent и normalized tools остаётся отдельной веткой для последующих правок.

### Contract `ProjectPlan`
- `ProjectPlan` должен быть пригоден для deterministic compilation без догадок на сервере.
- Корневой объект должен содержать:
  - `projectType`
  - `nodes`
  - `assumptions`
- Каждый узел должен поддерживать как минимум:
  - `nodeKey`
  - `title`
  - `parentNodeKey`
  - `kind` (`phase` | `task`)
  - `durationDays`
  - `dependsOn`
- Ограничения schema для `v1`:
  - `nodeKey` обязателен и уникален в рамках одного `ProjectPlan`;
  - `title` обязателен и не может быть placeholder-именем;
  - `durationDays` обязателен, целое число, `>= 1`;
  - `phase` может быть только контейнером и должна иметь хотя бы одного дочернего узла после compile cleanup;
  - `task` может быть как leaf, так и контейнером только если это явно разрешено планом; по умолчанию считать `task` leaf-узлом;
  - `parentNodeKey` для верхнеуровневых фаз отсутствует;
  - верхнеуровневый `task` в `v1` не допускается: все рабочие задачи должны находиться внутри фазы;
  - ссылки в `dependsOn` могут указывать только на существующие `nodeKey`.
- Зависимости должны поддерживать:
  - `dependencyType` (`FS` / `SS` / `FF` / `SF`)
  - `lagDays`
- Даже если planning-модель в `v1` почти всегда будет генерировать только `FS` без лагов, schema и compiler должны поддерживать dependency type / lag сразу.
- Server defaults:
  - если тип зависимости не указан, использовать `FS`
  - если лаг не указан, использовать `0`
  - если `assumptions` отсутствует, подставлять пустой массив.

## Проверки

### Тесты маршрутизации
- Пустой проект + broad generation request → `initial_generation` с двухэтапным pipeline.
- `Построй график строительства детского садика` → НЕ уходит в deterministic template path.
- Обычные edit-запросы → НЕ получают двухэтапный routing.
- Расплывчатый запрос → НЕ порождает уточняющих вопросов, а идёт в генерацию.
- `initial_generation` выбирает expensive model.
- Обычные mutation-запросы выбирают cheap model.

### Тесты planning-этапа
- Object type корректно попадает в server brief.
- Доменный справочник подмешивается в prompt для распознанного типа объекта.
- Для нераспознанного типа — подмешивается generic construction reference с default interpretation `частный жилой дом`.
- Output planning-этапа — валидный structured format (`ProjectPlan` с устойчивыми `nodeKey`, иерархией, зависимостями и lag metadata).
- Верхнеуровневые `task`-узлы и неуникальные `nodeKey` отвергаются schema-валидацией до compile.

### Тесты quality gate (между этапами)
- План без иерархии → repair.
- План с generic naming → repair.
- Предметный план с нормальной иерархией → принят, идёт на execution.
- После 1 repair-цикла план всё ещё слабый → принять лучший вариант, не зацикливаться.

### Тесты deterministic execution
- Утверждённый план → корректный `create_tasks_batch`, соответствующий плану.
- Внутренние `nodeKey` корректно мапятся в реальные task IDs.
- Родители и дети создаются в одном batch без поломки иерархии.
- Зависимости между сиблинговыми и межфазными задачами корректно компилируются.
- Зависимости с `FS/SS/FF/SF` и `lagDays` корректно компилируются или явно ограничиваются правилами compiler.
- Компиляция использует текущую дату сервера, business mode и календарь РФ по умолчанию.
- Parent phase даты формируются только rollup'ом от детей.
- Частично некомпилируемый план → compiler коммитит максимально полный валидный partial schedule, если результат остаётся предметным.
- Критично некомпилируемый план → controlled error, не молчаливый переход в другой flow.

### Тесты логов
- В логах видны: route, выбранный model tier, план, quality verdict, compile verdict, acceptance reason.
- В логах видны причины partial cleanup: какие узлы/связи были отброшены и почему.
- По одному run можно восстановить полный путь генерации.

### Ручные сценарии
- `Построй типичный график строительства`
- `Построй график строительства детского садика`
- `Построй график ремонта офиса 300 м2`
- `Построй график строительства частного дома из газобетона`
- `Построй график` (минимальный запрос — система справляется без уточнений)
- Для каждого сравнивать: предметность, иерархию, реалистичность последовательности, соответствие доменному справочнику и итоговый "вау-эффект".
- Отдельно сравнить latency и стабильность против текущего MCP-agent mutation flow.
- Для `Построй график` отдельно сравнить два варианта интерпретации: абстрактный generic construction project vs default `частный жилой дом`, и зафиксировать более сильный первый опыт.

## Допущения
- Для initial generation допустим latency 10–20 секунд.
- Planning-этап идёт на сильной модели, execution выполняется сервером детерминированно.
- Система не задаёт уточняющих вопросов; расплывчатый запрос обрабатывается с максимальной опорой на доменную экспертизу.
- После слабого плана — 1 внутренний repair, а не запрос к пользователю.
- Текущие deterministic-изменения с generic construction template откатываются.
- Доменный справочник — это NOT curated templates. Это compact reference для направления модели, а не готовые списки задач.
- Curated templates — отдельная фаза, не смешивать с AI-first архитектурой.
- Для `v1` initial generation ограничен construction / repair доменом; остальные домены остаются на текущем общем flow до отдельного расширения.
- Routing по моделям конфигурируется через env на сервере; выбор модели делается до старта конкретного SDK/agent run.
- Если `cheap_model` не задана, обычные правки используют ту же модель, что и initial planning.
- Для расплывчатого строительного запроса default interpretation — `частный жилой дом`, а не абстрактный generic объект.
- При compile failure partial-but-valuable результат предпочтительнее пустой ошибки, если он остаётся предметным и безопасным.
