# Рефакторинг AI-генерации стартового графика

## Кратко
Нужно вернуть initial generation к AI-first подходу и убрать текущий deterministic fast path для широких запросов в пустом проекте. Целевой режим: при первом запросе на построение графика система использует более сильную модель, подмешивает компактный серверный brief по типу объекта, делает 1–2 внутренних repair-цикла при слабом результате и принимает только такой график, который даёт предметный "вау-эффект", а не просто валидную мутацию.

## Ключевые изменения
- Убрать из серверного пайплайна перехват broad-запросов через `initial_schedule_template`.
- Удалить текущую логику, которая по широкому запросу в пустом проекте сразу создаёт generic construction template.
- Не заменять её новыми regexp-ветками для контентной генерации.
- Быстрые deterministic path допустимы позже только для узких, явно безопасных кейсов, где они не подменяют продуктовый AI-опыт.

- Ввести отдельный режим `initial_generation` для пустого проекта.
- Условие входа: проект пустой и запрос относится к широкому построению графика.
- Этот режим должен отличаться от обычного mutation flow не только intent-классификацией, но и orchestration-логикой, prompt strategy, quality gate и model routing.

- Добавить серверный AI-brief перед вызовом модели.
- Brief должен быть коротким и структурным, не в виде готового шаблона задач.
- В brief включать:
  - предполагаемый тип объекта из запроса
  - масштабные сигналы из текста пользователя
  - ожидание на предметность иерархии
  - требование создать не фрагмент, а стартовый каркас проекта
  - требование избегать generic filler вроде `Работа 1/2/3`
- Brief не должен жёстко навязывать список работ; он должен направлять модель, а не заменять её.

- Ввести отдельную маршрутизацию модели для первого broad-запроса в пустом проекте.
- Первый генеративный ход должен идти на более сильной модели.
- Обычные последующие правки могут остаться на более дешёвой модели.
- Routing должен опираться на класс запроса и состояние проекта, а не на список фраз.

- Перестроить retry в quality-driven repair loop.
- Попытка 1: обычная AI-генерация через нормализованные mutation tools.
- Если не было валидной мутации, либо получился слишком бедный/generic график, запускать repair attempt.
- В repair prompt передавать не абстрактное "попробуй ещё раз", а структурную критику:
  - недостаточная предметность под тип объекта
  - слишком маленький охват
  - слабая иерархия
  - отсутствие логичной последовательности/зависимостей
  - generic naming
  - неверная работа с ID / invalid mutation flow
- Ограничить внутренний repair 1–2 циклами в рамках целевого latency 10–20 секунд.

- Добавить quality gate поверх mutation verification.
- Сейчас система в основном проверяет, была ли мутация.
- Нужно отдельно оценивать продуктовое качество результата initial generation.
- Минимальный порог принятия:
  - график не тривиальный по объёму
  - есть несколько предметных фаз
  - есть иерархия, а не плоский список
  - названия специфичны для объекта
  - есть хотя бы базовая логика последовательности/зависимостей там, где это ожидаемо
- Если мутация валидна технически, но график слабый продуктово, результат не принимать сразу, а отправлять в repair.

- Переписать prompt contract именно для broad initial generation.
- Явно разделить в prompt два режима: обычное изменение проекта и стартовая генерация пустого проекта.
- Для стартовой генерации требовать:
  - стартовый skeleton проекта, а не `small fragment`
  - предметные этапы под тип объекта
  - чистую иерархию
  - реалистичную последовательность
  - отсутствие placeholder naming
- Не использовать fallback-инструкции, которые толкают модель к safe minimal output для broad generation.

- Усилить логирование качества AI-цикла.
- Логировать:
  - какой route выбран
  - какой model tier выбран
  - какой object type был выведен
  - почему был запущен repair
  - почему результат принят или отклонён
- Логи должны позволять разбирать не только "не было mutation", но и "mutation была, но график слабый".

## Интерфейсы и внутренняя структура
- Добавить явный внутренний тип/флаг режима генерации, например `initial_generation`, вместо неявного смешения через `mutationRequested` и `simpleMutationRequested`.
- Добавить типизированные структуры для:
  - generation brief
  - quality verdict
  - repair reason
  - model routing decision
- Не добавлять новые пользовательские MCP-tools на этом этапе.
- Вся реализация должна оставаться поверх текущих нормализованных mutation tools.

## Проверки
- Тесты маршрутизации:
  - пустой проект + broad generation request идёт в `initial_generation`
  - запрос вроде `Построй график строительства детского садика` не уходит в deterministic template path
  - обычные edit-запросы не получают special first-turn routing

- Тесты brief/prompt:
  - object type корректно попадает в server brief
  - prompt для initial generation требует hierarchy + object-specific phases
  - repair prompt содержит конкретную критику качества результата

- Тесты quality gate:
  - текстовый ответ без мутации уходит в repair
  - валидная, но generic генерация уходит в repair
  - предметный график с нормальной иерархией принимается без лишнего repair

- Тесты логов:
  - в логах видны route, model tier, repair reason, acceptance reason
  - по одному run можно восстановить, почему initial generation была слабой или успешной

- Ручные сценарии:
  - `Построй типичный график строительства`
  - `Построй график строительства детского садика`
  - `Построй график ремонта офиса 300 м2`
  - `Построй график строительства частного дома из газобетона`
  - Для каждого сравнивать предметность, иерархию, реалистичность последовательности и итоговый "вау-эффект".

## Допущения
- Для первого широкого построения пустого проекта допустим latency 10–20 секунд.
- Для initial generation допустимо использовать более сильную модель, чем для обычных правок.
- После первой слабой попытки система делает внутренний repair, а не сразу просит пользователя уточнять запрос.
- Текущие deterministic-изменения с generic construction template нужно откатить в рамках этого рефакторинга.
- Явные curated templates можно делать позже отдельной фазой, но не смешивать их с этой AI-first архитектурой.
