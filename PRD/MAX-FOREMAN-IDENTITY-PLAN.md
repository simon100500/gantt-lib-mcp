# Workflow идентификации прораба в мини-приложении MAX

## Кратко

- Целевой сценарий: первое корректное открытие MAX-ссылки привязывает ссылку факта к проверенному `MAX user.id`; дальше вносить факт по этой ссылке может только этот MAX-аккаунт.
- Текущий формат ссылки сохраняем: `https://max.ru/<bot>?startapp=<fact_slug>`.
- Для доверенной идентификации используем `window.WebApp.initData`, но проверяем его только на сервере через `MAX_BOT_TOKEN`.
- `initDataUnsafe` можно использовать только для UI, но не для авторизации.

## Что меняем

- На сервере добавить проверку MAX launch data:
  - env `MAX_BOT_TOKEN`;
  - HMAC-SHA256 проверка `WebAppData` / `window.WebApp.initData`;
  - отклонять устаревший `auth_date`, например старше 1 часа.
- Расширить модель токена факта:
  - `maxUserId`;
  - `maxBoundAt`;
  - `maxLastSeenAt`;
  - `maxUsername`, `maxFirstName`, `maxLastName`, `maxPhotoUrl`;
  - `maxPlatform`, `maxVersion`.
- Расширить журнал закрытия дня:
  - сохранять `actorMaxUserId`;
  - `actorMaxUsername`;
  - `actorDisplayName`.
- В `packages/fact`:
  - отправлять `window.WebApp.initData` в заголовке `X-Max-WebApp-Data` во все fact-запросы;
  - сохранить текущий парсинг `startapp`, `WebAppStartParam`, `?token=`;
  - без валидного MAX initData не разрешать запись факта.
- В основном UI:
  - показывать статус ссылки: `Не привязана`, `Привязана: <имя/id MAX>`, `Открыта другим MAX аккаунтом`;
  - добавить кнопку `Сбросить привязку` для владельца/редактора проекта.

## Workflow

- Руководитель создает ссылку "Ссылка прорабу" и отправляет ее в MAX.
- Прораб открывает ссылку в MAX.
- Мини-приложение получает `start_param` со slug токена и отправляет на сервер slug + `initData`.
- Сервер проверяет подпись MAX:
  - если токен еще не привязан, сохраняет `MAX user.id` как владельца этой fact-ссылки;
  - если токен уже привязан к этому же `MAX user.id`, разрешает загрузку и запись;
  - если токен привязан к другому `MAX user.id`, возвращает `403`;
  - если MAX-данные отсутствуют или подпись неверная, запрещает запись.
- Каждая отправка факта повторно проверяет MAX initData и пишет автора в журнал факта.

## Дополнительные фишки MAX

- Нативная кнопка "Назад" и подтверждение закрытия, чтобы прораб случайно не потерял ввод.
- Haptics после успешной отметки факта.
- Share/open link для пересылки ссылки или открытия связанных материалов.
- QR scanner для сценария "прораб сканирует QR на объекте".
- Secure storage / биометрия как v2, если захотим дополнительно подтверждать устройство.
- Запрос контакта/телефона как v2, если нужно сверять прораба не только по MAX ID, но и по номеру.

## Тесты

- Проверка HMAC: валидная подпись, неверная подпись, отсутствующий hash, устаревший `auth_date`.
- Первый вход привязывает токен к MAX user.
- Повторный вход того же MAX user разрешен.
- Другой MAX user получает `403`.
- Запись факта без валидного MAX initData запрещена.
- После сброса привязки токен можно привязать заново.
- В журнале факта сохраняется MAX-автор записи.

## Принятые допущения

- Выбранный уровень идентификации: `MAX ID + привязка`.
- Телефон и биометрия не входят в v1.
- Текущие обычные fact-ссылки можно оставить для просмотра/отладки, но запись факта в MAX-сценарии должна требовать проверенный MAX initData.
