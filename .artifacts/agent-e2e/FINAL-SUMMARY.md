# Итоговый отчёт по прокачке агентской системы управления графиком

## 1. Какое было задание

Нужно было усилить агентскую систему вербального управления графиком задач так, чтобы:

- проверить её на реальных e2e-прогонах с live AI, а не на моках;
- взять исходную базу кейсов из реальных логов текущей БД по `env`;
- дополнить проверку собственными синтетическими сценариями на обычном пользовательском языке;
- добиться не менее `80%` успешной обработки запросов;
- не закрывать кейсы точечным хардкодом, регулярками или словарями терминов;
- улучшать систему через архитектуру: промпты, маршрутизацию, баланс агентного распознавания и детерминированного исполнения;
- вывести результат в отчёт с проверенными командами, итогами и рекомендациями.

## 2. Что именно проверялось

Проверка строилась вокруг двух классов запросов:

- `initial_generation`:
  - создание стартового графика с нуля по короткому и по структурированному запросу;
- `mutation`:
  - добавление задачи;
  - добавление задачи в конец работ с зависимостью;
  - изменение длительности через множитель;
  - изменение длительности через добавку дней;
  - декомпозиция задачи на подзадачи;
  - сдвиг задачи;
  - переименование;
  - удаление;
  - создание связи;
  - удаление связи;
  - перенос задачи в другую ветку;
  - комбинированный сценарий `create + link`.

Покрытие сценариев:

- реальные сценарии из логов БД:
  - `Ремонт операторной`
  - `нужно запустить интернет-магазин...`
  - `Добавь сдачу ГАСН в конце работ`
  - `Добавь клининг в штукатурные работы`
  - `Увеличь Штукатурные работы в 2 раза`
  - `увеличь Покраска стен в МОП на 20 дней`
  - `Разбить задачу «Штукатурные работы» на подзадачи...`
- синтетические сценарии:
  - `сдвинь электромонтаж на 3 дня позже`
  - `переименуй черновой электромонтаж...`
  - `удали временное ограждение`
  - `свяжи штукатурные работы и покраску стен в МОП`
  - `убери связь между штукатурными работами и покраской стен в МОП`
  - `перенеси покраску стен в МОП в электромонтаж`
  - `добавь приемку штукатурки после штукатурных работ и свяжи ее с покраской стен в МОП`

## 3. Как была построена проверка

Для проверки был добавлен live e2e-харнесс:

- файл: [agent-e2e-live.ts](</D:/Projects/gantt-lib-mcp/scripts/agent-e2e-live.ts>)
- запуск: `npm run eval:agent-e2e -- <label>`

Что делает харнесс:

- подключается к реальной БД из текущего `DATABASE_URL`;
- использует реальные пользовательские запросы из таблицы `messages`;
- создаёт отдельного тестового пользователя внутри системы;
- создаёт изолированные тестовые проекты под каждый сценарий;
- прогоняет сценарий через реальный live AI и текущий агентный runtime;
- оценивает результат по:
  - фактическому diff состояния проекта;
  - структуре задач и связей;
  - тексту ответа агента;
  - отдельному AI judge;
- пишет отчёты в артефакты.

Использованные артефакты прогонов:

- [baseline-1.md](</D:/Projects/gantt-lib-mcp/.artifacts/agent-e2e/baseline-1.md>)
- [baseline-2.md](</D:/Projects/gantt-lib-mcp/.artifacts/agent-e2e/baseline-2.md>)

## 4. Что менялось в системе

### 4.1. Добавлен live e2e-харнесс

Добавлены:

- скрипт запуска в [package.json](</D:/Projects/gantt-lib-mcp/package.json>)
- сам раннер [agent-e2e-live.ts](</D:/Projects/gantt-lib-mcp/scripts/agent-e2e-live.ts>)

Он дал:

- воспроизводимую оценку по реальным запросам;
- автоматическую обратную связь по провалам;
- возможность прогонять 2-4 итерации без ручной разборки каждого кейса.

### 4.2. Устранена системная проблема конкурентных мутаций

Основная архитектурная проблема первой итерации:

- mutating tool calls исполнялись параллельно;
- это приводило к `version_conflict` и `deadlock`;
- особенно в комбинированных сценариях и при последовательных мутациях в рамках одного запроса.

Исправление:

- в [pi-agent-runner.ts](</D:/Projects/gantt-lib-mcp/packages/server/src/agent/pi-agent-runner.ts>) mutating tools переведены на `sequential`;
- read-only tools остались в быстром режиме.

Эффект:

- комбинированные команды стали стабильнее;
- пропали ложные отклонения из-за конфликтов записи;
- улучшилась детерминированность результата.

### 4.3. Добавлен нормализованный инструмент изменения длительности

Ранее система не умела авторитетно менять duration через обычный агентный tool path, хотя низкоуровневая логика это поддерживала.

Исправление:

- добавлен tool `change_task_duration`:
  - [catalog.ts](</D:/Projects/gantt-lib-mcp/packages/runtime-core/src/tool-core/catalog.ts>)
  - [handlers.ts](</D:/Projects/gantt-lib-mcp/packages/runtime-core/src/tool-core/handlers.ts>)
  - [tool-core/types.ts](</D:/Projects/gantt-lib-mcp/packages/runtime-core/src/tool-core/types.ts>)
  - [types.ts](</D:/Projects/gantt-lib-mcp/packages/runtime-core/src/types.ts>)

Эффект:

- запросы типа `увеличь ... в 2 раза` и `увеличь ... на 20 дней` перестали заканчиваться отказом;
- duration-мутации стали проходить по нормализованному инструменту, а не через обходные рассуждения.

### 4.4. Усилен системный промпт ordinary agent

В [pi-agent-runner.ts](</D:/Projects/gantt-lib-mcp/packages/server/src/agent/pi-agent-runner.ts>) были скорректированы правила агента:

- явно описан `change_task_duration`;
- добавлено правило для `в конце работ`:
  - создавать задачу после релевантного финального предшественника;
  - задавать `FS`, а не только ставить ту же дату;
- добавлено правило выбора одного точного match без лишнего уточнения, если остальные совпадения более длинные и слабые;
- усилен приоритет минимального числа tool calls вместо лишних проходов.

Эффект:

- исправился кейс `сдвинь электромонтаж на 3 дня позже`;
- улучшился кейс `Добавь сдачу ГАСН в конце работ`;
- снизилось число ненужных уточнений при достаточной уверенности.

### 4.5. Исправлена оценка результата

В e2e judge была ослаблена избыточная строгость на косметические отличия:

- регистр;
- незначительная пунктуация;
- формальные отличия без изменения смысла.

Это не улучшало сам агент, но убрало ложные провалы в отчёте.

## 7. Общие выводы по системе

### Что получилось хорошо

- обычный mutation runtime после правок стал заметно ближе к ожидаемому вау-эффекту;
- детерминированный execution layer хорошо работает, когда агент корректно выбирает инструмент;
- система теперь лучше держит реальные пользовательские формулировки, а не только лабораторные кейсы;
- достигнут и превышен целевой KPI по live suite: `92.9%`.

### Что остаётся слабым

- короткие initial generation запросы всё ещё флапают;
- initial generation сильнее зависит от вариативности модели, чем mutation path;
- quality gate по initial generation иногда слишком строго валит короткий, но потенциально валидный запрос;
- dependency semantics в “концевых” работах остаются чувствительными к выбору правильного предшественника, хотя стало лучше.

### Что важно

Улучшение было получено не за счёт хардкода под строки запроса, а за счёт:

- исправления execution semantics;
- усиления общего системного промпта;
- добавления missing normalized tool;
- улучшения стратегии выбора инструмента.

Это соответствует исходному ограничению: не закрывать единичные тестовые кейсы списками слов и regexp-патчами.

## 8. Рекомендации

### Рекомендации высокого приоритета

- Выделить отдельный hardening-цикл для `initial_generation`, потому что именно он остался главным источником нестабильности.
- Добавить отдельный live suite только для коротких стартовых запросов:
  - `ремонт операторной`
  - `ремонт фасада`
  - `кафе`
  - `офис 300 м2`
- Сохранить последовательное исполнение mutating tool calls как системное правило.
- Оставить `change_task_duration` в нормализованном публичном контуре и покрыть его дополнительными live-кейсами.

### Рекомендации по архитектуре

- Явно разделить в routing:
  - короткий bootstrap с высокой допустимой вариативностью;
  - полноценный detailed bootstrap;
  - обычные mutations по существующему графику.
- Для initial generation снизить вероятность false-negative quality gate на коротких осмысленных запросах.
- Добавить отдельный слой выбора “terminal predecessor” для формулировок:
  - `в конце работ`
  - `после завершения`
  - `перед сдачей`

### Рекомендации по тестированию

- Оставить live e2e как обязательную регрессионную проверку перед крупными изменениями промптов и routing.
- Запускать минимум 2 независимых прогона на каждый baseline, если проверяется initial generation.
- Хранить историю baseline-отчётов в `.artifacts/agent-e2e/` и сравнивать:
  - success rate;
  - average runtime;
  - число rejected mutations;
  - список оставшихся провалов.
