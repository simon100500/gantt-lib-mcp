# Handoff: тестирование отдельных user-cases через live e2e методику

## Что это за артефакт

Этот handoff нужен, чтобы новый агент или инженер мог быстро взять один конкретный пользовательский кейс, прогнать его через текущую live e2e методику и понять:

- как оформить кейс в сценарий;
- какой fixture выбрать;
- как задать expectation;
- как запускать прогон точечно;
- как интерпретировать провал;
- как понять, что именно надо чинить: prompt, routing, tool surface, execution layer или evaluator.

Основные исходники:

- методика и общие результаты: [FINAL-SUMMARY.md](/D:/Projects/gantt-lib-mcp/.artifacts/agent-e2e/FINAL-SUMMARY.md)
- лучший подтвержденный baseline: [baseline-3.md](/D:/Projects/gantt-lib-mcp/.artifacts/agent-e2e/baseline-3.md)
- live harness: [agent-e2e-live.ts](/D:/Projects/gantt-lib-mcp/scripts/agent-e2e-live.ts)

## Текущий статус методики

Сейчас проверка уже работает end-to-end:

- используется реальная БД из текущего `.env`;
- реальные кейсы можно брать из таблицы `messages`;
- под каждый сценарий поднимается изолированный тестовый проект;
- используется реальный live AI, а не моки;
- результат проверяется через structural diff и AI judge;
- лучший зафиксированный результат: `92.9%` (`13/14`) на `baseline-3`.

Ограничение методики:

- слабое место остается в `initial_generation`, особенно на очень коротких запросах;
- mutation-контур уже достаточно стабилен и удобен для прицельной регрессии по отдельным кейсам.

## Быстрый операторский сценарий

Если нужно проверить один новый пользовательский кейс, делай так:

1. Возьми формулировку пользователя из логов или от PM/support.
2. Классифицируй кейс как `mutation` или `initial_generation`.
3. Подбери fixture, который ближе всего к реальному состоянию проекта.
4. Добавь сценарий в `SCENARIOS` в [agent-e2e-live.ts](/D:/Projects/gantt-lib-mcp/scripts/agent-e2e-live.ts).
5. Сформулируй `expectation` только через наблюдаемый результат, без завязки на внутренние догадки агента.
6. Запусти отдельный label для этого тестового захода.
7. Открой `.json` и `.md` отчет в `.artifacts/agent-e2e/`.
8. Определи тип провала и только потом выбирай способ исправления.
9. После локального фикса перепроверь кейс и затем прогоняй полный baseline.

## Как брать кейс из логов

Для log-derived сценария нужны две вещи:

- текст пользовательского запроса;
- `requestContextId`, чтобы сохранить привязку к исходному кейсу.

В сценарии это выглядит так:

```ts
{
  id: 'log-some-case',
  origin: 'log',
  description: 'Short explanation of the case',
  userMessage: 'Текст пользовательского запроса',
  fixture: 'renovation',
  expectation: {
    route: 'mutation',
    createdTaskNames: ['Новая задача'],
  },
  logRequestContextId: 'uuid-from-messages',
}
```

`logRequestContextId` не управляет исполнением. Он нужен для трассировки источника в отчете и для удобства последующего разбора.

## Как выбрать fixture

В текущем harness есть три режима:

- `empty`
- `closeout`
- `renovation`

Когда что использовать:

- `empty`
  Используй для `initial_generation`, когда пользователь просит построить график с нуля.

- `closeout`
  Используй для сценариев про завершение проекта, сдачу, ввод в эксплуатацию, финальные этапы, привязку "в конце работ".

- `renovation`
  Используй для большинства mutation-кейсов: добавить задачу, сдвинуть, переименовать, удалить, изменить длительность, перенести в ветку, добавить или убрать связь.

Когда нужно добавлять новый fixture:

- кейс требует особой структуры проекта, которой нет в `closeout` и `renovation`;
- корректность сценария зависит от нескольких веток, milestone-узлов или иной dependency topology;
- текущий fixture заставляет тест проверять не тот риск, который реально интересует.

Если нужен новый fixture, ориентир простой: он должен моделировать форму проекта, а не конкретную фразу пользователя.

## Как задавать expectation

Expectation должен описывать наблюдаемый итог, а не внутренний план агента.

Поддерживаемые типы ожиданий сейчас:

- `route`
- `minTaskCount`
- `createdTaskNames`
- `absentTaskNames`
- `renamedTask`
- `durationTask`
- `shiftedTask`
- `dependency`
- `parentTask`

Примеры:

Создание задачи:

```ts
expectation: {
  route: 'mutation',
  createdTaskNames: ['Сдача ГАСН'],
}
```

Удаление:

```ts
expectation: {
  route: 'mutation',
  absentTaskNames: ['Временное ограждение'],
}
```

Изменение длительности:

```ts
expectation: {
  route: 'mutation',
  durationTask: {
    name: 'Штукатурные работы',
    minMultiplier: 1.8,
  },
}
```

Изменение родителя:

```ts
expectation: {
  route: 'mutation',
  parentTask: {
    taskName: 'Покраска стен в МОП',
    parentName: 'Электромонтаж',
  },
}
```

Правила формулировки expectation:

- проверяй минимально достаточный эффект;
- не делай expectation слишком хрупким;
- не завязывайся на косметику ответа;
- если кейс комбинированный, проверяй оба эффекта;
- если кейс про `initial_generation`, не пытайся жестко диктовать весь WBS, лучше использовать `minTaskCount` и здравую оценку judge.

## Как добавить сценарий

Сценарии сейчас зашиты в массив `SCENARIOS` в [agent-e2e-live.ts](/D:/Projects/gantt-lib-mcp/scripts/agent-e2e-live.ts).

Минимальный шаблон для нового mutation-case:

```ts
{
  id: 'synthetic-add-final-inspection',
  origin: 'synthetic',
  description: 'Add final inspection after closeout',
  userMessage: 'добавь финальную инспекцию после сдачи объекта',
  fixture: 'closeout',
  expectation: {
    route: 'mutation',
    createdTaskNames: ['Финальная инспекция'],
  },
}
```

После добавления сценария скрипт сам:

- создаст тестовый проект;
- зальет fixture;
- прогонит агент;
- соберет diff;
- вызовет judge;
- сохранит `.json` и `.md` отчет.

## Как запускать

Базовый запуск:

```bash
npm run eval:agent-e2e -- case-check-01
```

Результаты будут здесь:

- `.artifacts/agent-e2e/case-check-01.json`
- `.artifacts/agent-e2e/case-check-01.md`

Сейчас harness прогоняет весь массив `SCENARIOS`. Это значит, что для точечной проверки одного кейса пока приходится:

- либо временно комментировать остальные сценарии;
- либо временно оставлять в массиве только нужный;
- либо заводить отдельный временный label под малый набор кейсов.

Это рабоче, но это одно из главных мест для улучшения методики.

## Как читать результат

В `.md` отчете по сценарию смотри:

- `Status`
- `Route`
- `Tool calls`
- `accepted mutations`
- `rejected mutations`
- `Assistant response`
- `Judge`
- `Created/Deleted/Changed tasks`
- `Notes`

В `.json` отчете полезны поля:

- `success`
- `structuralPass`
- `hardError`
- `judge.verdict`
- `judge.matchedRequest`
- `judge.hasError`
- `notes`
- `createdTaskNames`
- `deletedTaskNames`
- `changedTaskNames`

Практическое чтение:

- `structuralPass=false`
  Почти всегда агент реально не сделал ожидаемое изменение или сделал не то.

- `acceptedMutations=0` при `route=mutation`
  Обычно это routing/prompt/tool-selection проблема.

- `rejectedMutations>0`
  Обычно это execution conflict, invalid tool payload или sequencing issue.

- `judge=failure`, но `structuralPass=true`
  Проверяй evaluator/judge-слой и качество описания expectation.

- `route mismatch`
  Сначала смотри route selection и системный prompt, а не evaluator.

## Как классифицировать провал

Ниже рабочая схема разбора.

### 1. Prompt / routing issue

Признаки:

- выбран не тот route;
- агент уходит в уточнения при достаточной определенности;
- агент понимает запрос не как то действие;
- mutation-кейс уходит в пустое рассуждение без мутаций.

Типичные решения:

- усилить routing criteria;
- уточнить системные инструкции ordinary agent;
- добавить обобщенное правило выбора инструмента;
- убрать неоднозначность между shift, duration, move, create-link.

### 2. Tool-surface gap

Признаки:

- запрос понятен, но нет удобного нормализованного tool path;
- агент вынужден делать обходной маневр;
- один и тот же класс кейсов валится стабильно на похожей семантике.

Типичные решения:

- добавить новый публичный tool;
- нормализовать входной контракт существующего tool;
- расширить tool schema под частый реальный паттерн.

`change_task_duration` был именно таким исправлением.

### 3. Execution conflict / sequencing issue

Признаки:

- есть `rejectedMutations`;
- есть `version_conflict` или признаки гонок;
- комбинированный кейс ломается только при нескольких мутациях подряд;
- одиночные шаги проходят, а составной сценарий нет.

Типичные решения:

- перевести mutating calls в `sequential`;
- уменьшить параллелизм на записи;
- разделить read-only и mutating execution semantics;
- стабилизировать порядок применения связанных изменений.

### 4. Evaluator issue

Признаки:

- фактически проект изменился правильно;
- diff выглядит разумно;
- но сценарий помечен как fail из-за косметики или слишком жесткого expectation.

Типичные решения:

- ослабить judge на регистр и косметику;
- переписать expectation так, чтобы он проверял смысл;
- сохранять больше отладочной информации judge.

### 5. Fixture problem

Признаки:

- запрос сам по себе нормальный, но выбранный fixture не соответствует реальной среде кейса;
- agent делает разумное действие, но не в той структуре проекта;
- expectation спорит с самим fixture.

Типичные решения:

- выбрать другой fixture;
- добавить новый fixture;
- уменьшить искусственность тестовой сцены.

## Рекомендуемый разбор одного кейса

Для нового кейса используй такой цикл:

1. Возьми исходную формулировку пользователя без переписывания.
2. Реши, это `initial_generation` или `mutation`.
3. Подбери fixture, близкий к реальному проекту пользователя.
4. Задай один основной expectation и только потом добавляй вторичные.
5. Прогони кейс.
6. Если fail, сначала открой `notes`, `assistantResponse` и `changedTaskNames`.
7. Определи класс провала.
8. Делай архитектурный фикс, а не phrase-specific patch.
9. Прогони кейс повторно.
10. Затем прогони полный baseline, чтобы убедиться, что нет регрессии.

## Что совершенствовать в методике

Ниже улучшения с наибольшей практической отдачей.

### Высокий приоритет

- Вынести `SCENARIOS` из [agent-e2e-live.ts](/D:/Projects/gantt-lib-mcp/scripts/agent-e2e-live.ts) в отдельный `json` или `ts` модуль.
- Добавить CLI-фильтр вида `--scenario <id>` для запуска одного кейса.
- Добавить CLI-фильтр вида `--origin log|synthetic` и `--route mutation|initial_generation`.
- Разделить suite на `mutation` и `initial_generation`, чтобы не смешивать два сильно разных класса нестабильности.
- Добавить отдельный retry policy для коротких `initial_generation` кейсов, чтобы измерять флап явно, а не путать его с полной поломкой.

### Средний приоритет

- Расширить библиотеку fixtures:
  - строительство;
  - IT/launch;
  - интерьер/ремонт;
  - closeout-heavy проект;
  - проект с несколькими параллельными ветками и milestone.
- Сохранять сырые judge payloads в артефакты.
- Сохранять route-selection evidence по каждому кейсу.
- Вынести common helpers по expectation в отдельный модуль, чтобы сценарии было проще читать и расширять.

### Низкий, но полезный приоритет

- Добавить компактный CSV/summary-индекс по всем baseline-отчетам.
- Добавить историческое сравнение success rate и median runtime по label.
- Добавить пометку flakiness per scenario, если один и тот же кейс прогоняется несколько раз.

## Как улучшать саму методику, а не только систему

Если цель именно в зрелости процесса тестирования, то методику стоит двигать в три стороны:

1. От ручного массива сценариев к декларативному scenario catalog.
2. От разового pass/fail к повторяемому измерению flakiness по каждому кейсу.
3. От общего итогового отчета к диагностическим артефактам, которые сразу показывают класс поломки.

Практический следующий шаг:

- первым делом добавить `--scenario <id>`;
- затем вынести сценарии в отдельный файл;
- затем начать хранить raw judge + route diagnostics.

Это даст быстрый выигрыш именно для задачи "тестировать отдельные user cases по методике", не ломая текущий harness.

## Что не делать

- Не подгонять систему под одну фразу через regexp или словарь синонимов.
- Не делать expectation чрезмерно детальным там, где важен только один смысловой эффект.
- Не чинить evaluator, если по факту агент действительно сделал неверную мутацию.
- Не оценивать mutation-кейсы на `empty` fixture.
- Не смешивать проблемы `initial_generation` и `mutation` в одном выводе без разделения.
