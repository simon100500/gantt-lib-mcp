# Milestone v2.0: PostgreSQL Migration

**Status:** ✅ SHIPPED 2026-03-17
**Phases:** 15-16
**Total Plans:** 6

## Overview

Replace SQLite with PostgreSQL + Prisma ORM for production scalability with multiple concurrent users. Prisma provides type-safe database access, migration management, and connection pooling.

## Phases

### Phase 15: Prisma Setup

**Goal**: PostgreSQL database with Prisma ORM is ready for development
**Depends on**: Nothing (first phase of v2.0)
**Plans**: 2 plans

**Requirements:** DB-01, DB-02, DB-03, DB-04, DB-05, POOL-01, POOL-02, POOL-03

**Success Criteria:**
1. Prisma schema defines all existing tables with proper relationships
2. Prisma client generates successfully and is accessible from packages/mcp and packages/server
3. DATABASE_URL connects to PostgreSQL with connection pooling
4. Prisma migrations run successfully
5. Connection pool settings appropriate for container constraints

Plans:
- [x] 15-01: Prisma schema and client singleton with connection pooling
- [x] 15-02: Initial migration execution and database verification

**Details:**
- Prisma schema: 10 models (User, Project, Session, OtpCode, Task, Dependency, Message, ShareLink, TaskRevision, TaskMutation)
- Prisma Client singleton with hot reload safety
- Connection pooling: connection_limit=10, pool_timeout=20s
- Migration: 20260313_init with PostgreSQL DDL
- Graceful shutdown on SIGTERM/beforeExit

---

### Phase 16: Services Layer

**Goal**: All database operations use Prisma-backed services instead of direct SQL
**Depends on**: Phase 15 (Prisma client and schema must exist)
**Plans**: 4 plans

**Requirements:** SVC-01, SVC-02, SVC-03, SVC-04, SVC-05, SVC-06, SVC-07

**Success Criteria:**
1. TaskService provides all CRUD operations using Prisma
2. ProjectService, AuthService, MessageService, DependencyService exist and use Prisma
3. Services shared between packages/mcp and packages/server (no duplicate code)
4. No raw SQL queries remain in service implementations
5. TypeScript types match Prisma-generated types

Plans:
- [x] 16-01: TaskService and DependencyService with Prisma CRUD operations
- [x] 16-02: AuthService and ProjectService with Prisma CRUD operations
- [x] 16-03: MessageService and service exports
- [x] 16-04: End-to-end verification and testing

**Details:**
- TaskService: create, update, delete, list, recalculateDates
- ProjectService: create, get, update, delete
- AuthService: OTP, sessions, JWT validation
- MessageService: CRUD for chat messages
- DependencyService: task dependency CRUD
- All services use Prisma client (no raw SQL)
- Services exported from packages/mcp for use by packages/server

## Milestone Summary

**Key Decisions:**
- Prisma schema location: packages/mcp/prisma/schema.prisma
- Prisma Client output: packages/mcp/dist/prisma-client
- Primary key type: uuid() for all tables
- Connection pool: connection_limit=10, pool_timeout=20s, connect_timeout=10s
- Foreign keys: onDelete: Cascade for all relations (except Task.parent_id with SetNull)
- Migration strategy: Prisma Migrate (not db push) for production history
- Services shared via package exports from packages/mcp

**Issues Resolved:**
- SQLite bottleneck for concurrent users resolved
- Type-safe database access achieved via Prisma-generated types
- Duplicate database code eliminated between packages/mcp and packages/server

**Technical Debt Incurred:**
- None — migration was clean, SQLite code fully replaced
