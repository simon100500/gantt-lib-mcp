## v3.0 Requirements — SHIPPED

MCP server refactoring: token economy, agent hardening, task hierarchy, conversation history, tool quality, Zustand frontend refactor, UI filters.

### Token Economy (TOKEN)

- [x] **TOKEN-01**: `get_tasks` supports compact mode with only essential fields (id, name, dates, parentId, progress) — ✅ Delivered Phase 17
- [x] **TOKEN-02**: `get_tasks` supports pagination with `limit` (default: 100) and `offset` (default: 0) parameters — ✅ Delivered Phase 17
- [x] **TOKEN-03**: `get_task` supports `includeChildren: boolean` (default: false) to avoid loading child tasks — ✅ Delivered Phase 17
- [x] **TOKEN-04**: Conversation history limited to 20 messages with truncation notice when exceeded — ✅ Delivered Phase 17

### Hardening (HARD)

- [x] **HARD-01**: Agent has max session turns limit of 20 — ✅ Delivered Phase 18
- [x] **HARD-02**: Agent has 2-minute timeout via AbortController — ✅ Delivered Phase 18
- [x] **HARD-03**: Agent excluded from direct file system and terminal tools (write_file, edit_file, run_terminal_cmd, run_python_code) — ✅ Delivered Phase 18

### Task Hierarchy (HIER)

- [x] **HIER-01**: `create_task` accepts `parentId?: string` parameter — ✅ Delivered Phase 19
- [x] **HIER-02**: `update_task` accepts `parentId?: string | null` parameter (null removes from parent) — ✅ Delivered Phase 19
- [x] **HIER-03**: `get_tasks` supports filtering by `parentId?: string | null` (null = root only, string = direct children) — ✅ Delivered Phase 19

### Conversation History (HIST)

- [x] **HIST-01**: New MCP tool `get_conversation_history` returns last N messages (limit: 20, max: 50) — ✅ Delivered Phase 20
- [x] **HIST-02**: New MCP tool `add_message` records assistant message to project chat history — ✅ Delivered Phase 20

### Tool Quality (QUAL)

- [x] **QUAL-01**: All tool descriptions are semantic and dense with usage guidance — ✅ Delivered Phase 21
- [x] **QUAL-02**: Error messages follow "what + why + what to do" pattern with actionable guidance — ✅ Delivered Phase 21

### Frontend Zustand Refactor (WEB-ZUSTAND)

- [x] **WEB-ZUSTAND-01**: Zustand is installed in `packages/web` and available to the frontend bundle — ✅ Delivered Phase 22
- [x] **WEB-ZUSTAND-02**: Foundational chat and UI frontend state lives in dedicated Zustand stores — ✅ Delivered Phase 22
- [x] **WEB-ZUSTAND-03**: Auth/session/project ownership moves into `useAuthStore` while `useAuth` remains a compatibility wrapper — ✅ Delivered Phase 22
- [x] **WEB-ZUSTAND-04**: Task collections and source selection move into `useTaskStore` — ✅ Delivered Phase 22
- [x] **WEB-ZUSTAND-05**: WebSocket task/chat side effects write directly into stores — ✅ Delivered Phase 22
- [x] **WEB-ZUSTAND-06**: `App.tsx` becomes a thin workspace router with dedicated workspace components — ✅ Delivered Phase 22
- [x] **WEB-ZUSTAND-07**: Toolbar and project-menu controls are driven by UI-store state instead of app-local orchestration — ✅ Delivered Phase 22

### UI Filters (FILTER)

- [x] **FILTER-01**: Filter state stored in useUIStore with fields: filterWithoutDeps, filterExpired, filterSearchText, filterDateFrom, filterDateTo — ✅ Delivered Phase 23
- [x] **FILTER-02**: Filter state persisted to localStorage with key 'gantt-filters' and loaded on mount — ✅ Delivered Phase 23
- [x] **FILTER-03**: GanttChart component accepts taskFilter prop of type TaskPredicate from gantt-lib — ✅ Delivered Phase 23
- [x] **FILTER-04**: Filter button in Toolbar opens popup with controls: checkboxes (without deps, expired), text input (search), date inputs (from, to), reset button — ✅ Delivered Phase 23
