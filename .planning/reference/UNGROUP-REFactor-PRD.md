# PRD: Удалить встроенный `ungroup` из `gantt-lib` и реализовать его в приложении через кастомное меню

## Статус
Draft

## Контекст
Сейчас в `gantt-lib` есть встроенная функция `Разгруппировать`, которая рендерится в меню строки задачи и поддерживается библиотечным API.

Для нашего приложения это плохая граница ответственности. `Ungroup` у нас не является простой UI-операцией. Это доменный сценарий, который:

- меняет иерархию задач
- удаляет только контейнер, а не всё поддерево
- чистит зависимости на удаляемую группу
- должен уважать наш command flow, optimistic state и серверную семантику

Даже если передавать `onUngroupTask`, сама идея встроенной операции в библиотеке создаёт лишнюю связность: библиотека начинает знать о сценарии, который на самом деле принадлежит приложению.

Целевое направление: убрать встроенный `ungroup` из `gantt-lib` полностью и реализовать его только в приложении через `taskListMenuCommands`.

## Проблема
Встроенный `ungroup` в библиотеке создаёт несколько архитектурных проблем:

- библиотека берёт на себя доменную операцию, а не универсальный UI primitive
- публичный API `gantt-lib` становится толще и менее универсальным
- остаётся риск скрытых конфликтов между библиотечной логикой и приложенческой семантикой удаления
- операция выглядит как часть общего продукта библиотеки, хотя реально зависит от конкретного приложения

Это особенно плохо для controlled integrations, где библиотека не должна предполагать, как именно устроены delete flow, hierarchy updates и server persistence.

## Цель
Сделать `ungroup` полностью приложенческой функцией и убрать её из built-in API `gantt-lib`.

## Не-цели
В этот рефактор не входят:

- переработка всех hierarchy-операций (`promote`, `demote`, `reorder`)
- редизайн стандартного task context menu
- полная переработка `useBatchTaskUpdate`
- изменение пользовательского UX текста `Разгруппировать`

## Ожидаемый результат
После рефактора:

- `gantt-lib` больше не содержит built-in `ungroup`
- `gantt-lib` не экспортирует и не документирует `onUngroupTask`
- приложение само добавляет `Разгруппировать` через `taskListMenuCommands`
- вся семантика ungroup живёт только в приложении
- операция не может случайно пересечься с общим delete flow библиотеки

## Product Principle
`gantt-lib` должен предоставлять универсальные UI-механизмы и low-level editing primitives, но не встраивать доменные составные операции, если они:

- комбинируют несколько изменений данных
- зависят от внешней persistence model
- имеют неоднозначную бизнес-семантику в разных приложениях

`Ungroup` относится именно к таким операциям.

## Scope: библиотека (`gantt-lib`)

1. Удалить built-in пункт меню `Разгруппировать`.
2. Удалить `onUngroupTask` из публичного API библиотеки.
3. Удалить внутреннюю реализацию ungroup и связанные fallback-сценарии.
4. Сохранить и поддержать extensibility через `taskListMenuCommands`, чтобы приложение могло добавить собственный пункт меню.
5. Обновить документацию библиотеки так, чтобы кастомные menu commands были официальным способом добавления приложенческих действий.

## Scope: приложение (`gantt-lib-mcp`)

1. Добавить пункт `Разгруппировать` через `taskListMenuCommands`.
2. Показ этого пункта должен происходить только для parent task, у которых есть direct children.
3. Клик по пункту должен вызывать только приложенческий `handleUngroupTask`.
4. Вся логика `ungroup` должна жить только в приложении.
5. Для auth mode операция должна идти через явный business flow.
6. Для local mode допускается локальное применение той же семантики.

## Функциональные требования

### FR1. Библиотека не содержит built-in ungroup
В `gantt-lib` больше не должно быть встроенного меню `Разгруппировать`, встроенного handler-а ungroup или публичного пропа `onUngroupTask`.

### FR2. Приложение добавляет ungroup как custom menu command
В `gantt-lib-mcp` пункт `Разгруппировать` должен добавляться через `taskListMenuCommands`.

### FR3. Семантика ungroup в приложении
Для parent task с direct children операция должна:

- удалить только parent
- поднять direct children на уровень `parent.parentId`
- не трогать grandchildren напрямую
- удалить зависимости вида `dep.taskId === removedParentId`

### FR4. Безопасность удаления
`Ungroup` не должен использовать такой flow, при котором delete parent приводит к удалению всего поддерева.

### FR5. Visibility rule
Пункт `Разгруппировать` должен быть доступен только тогда, когда задача является parent и у неё есть direct children.

### FR6. Controlled mode
В auth or controlled mode операция должна исполняться как явный бизнес-сценарий приложения.

### FR7. Local mode
В local mode визуальный и структурный результат должен совпадать с auth mode.

## Нефункциональные требования

- API `gantt-lib` должен стать проще, а не сложнее
- библиотека не должна включать спорную доменную семантику
- поведение приложения должно быть детерминированным
- кастомное меню должно позволять реализовать действие без форков библиотеки

## API изменения

### В `gantt-lib`
- удалить `onUngroupTask?: (taskId: string) => void`
- удалить built-in rendering menu item `Разгруппировать`
- удалить внутренний ungroup handler
- сохранить и документировать `taskListMenuCommands` как supported extension point

### В приложении
- пункт `Разгруппировать` добавляется через `taskListMenuCommands`
- handler приложения становится единственной точкой входа для ungroup
- обёртка [packages/web/src/components/GanttChart.tsx](D:/Projects/gantt-lib-mcp/packages/web/src/components/GanttChart.tsx) не должна содержать отдельный API для `onUngroupTask`

## Предлагаемая реализация

### Фаза 1. Library cleanup
- убрать built-in ungroup из `gantt-lib`
- удалить `onUngroupTask` из типов и реализации
- обновить тесты библиотеки

### Фаза 2. App migration
- добавить кастомный menu command `Разгруппировать` в `gantt-lib-mcp`
- привязать его к `handleUngroupTask`
- сохранить текущую бизнес-семантику ungroup на стороне приложения

### Фаза 3. Cleanup
- проверить, что больше нет остаточных ссылок на `onUngroupTask`
- при необходимости вынести hierarchy operations из `useBatchTaskUpdate` в отдельный модуль приложения

## Критерии приёмки

1. В `gantt-lib` отсутствует built-in `Разгруппировать`.
2. В `gantt-lib` отсутствует публичный `onUngroupTask`.
3. В приложении пункт `Разгруппировать` присутствует как custom menu command.
4. Пункт показывается только для parent task с direct children.
5. При выполнении ungroup удаляется только parent.
6. Direct children поднимаются на один уровень выше.
7. Grandchildren сохраняют свою текущую родительскую связь.
8. Зависимости на удаляемый parent удаляются.
9. Auth mode не удаляет subtree по delete cascade.
10. Local mode даёт тот же структурный результат.
11. Сборка и тесты целевых пакетов проходят.

## Тест-кейсы

- parent с двумя direct children, без зависимостей
- parent с `parentId`, чтобы проверить переподвеску детей к дедушке
- parent с child и grandchild, чтобы проверить, что grandchild не поднимается лишний раз
- task вне subtree, зависящий от parent
- parent без детей: пункт меню не показывается
- local mode: ungroup даёт правильную структуру
- auth mode: ungroup не вызывает cascade delete subtree
- библиотека: custom menu продолжает поддерживать приложенческие действия

## Риски

- это более явный breaking change для `gantt-lib`
- если у библиотеки есть внешние потребители, им понадобится migration path
- если кастомные menu commands недостаточно гибкие, в приложении может понадобиться небольшой доработанный extension point

## Решение по рискам

- оформить изменение как intentional breaking API cleanup
- задокументировать migration path: "используйте `taskListMenuCommands` вместо built-in ungroup"
- перед удалением проверить, что кастомное меню поддерживает все нужные условия видимости и обработчики

## Migration Note
Потребители `gantt-lib`, которым нужен `ungroup`, должны:

1. добавить пункт меню через `taskListMenuCommands`
2. самостоятельно реализовать handler ungroup в приложении
3. не ожидать built-in поведения от библиотеки

- PR в `gantt-lib-mcp`, который реализует `Разгруппировать` через `taskListMenuCommands`
- тесты на библиотечный extension point и приложенческую семантику
- короткая migration note для потребителей библиотеки
