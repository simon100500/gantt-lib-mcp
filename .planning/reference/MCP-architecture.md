## 14. Где должен жить MCP server
Лучше **отдельным сервисом**, а не внутри основного API процесса.

**Почему:**
-   проще изолировать AI-логику
-   проще масштабировать отдельно
-   не мешает основному CRUD/realtime
-   можно независимо менять toolset

Но он не должен напрямую лазить в БД как попало.

**Лучше так:**
```
MCP Server -> calls App API / Command API / Domain Tools
```
Или через внутренний service layer.

## 15. Что должен уметь MCP server
По сути он должен быть **tool gateway** для AI.

**Например tools:**
-   `read_project` — Читает структуру проекта
-   `list_tasks` — Список задач с фильтрами
-   `create_task` — Создать задачу
-   `update_task` — Поменять сроки/название/зону
-   `create_dependency` — Создать связь
-   `repeat_by_zones` — Размножить по этажам/секциям
-   `recalculate_schedule` — Пересчитать
-   `explain_task` — Объяснить, почему так
-   `import_work_items` — Создать задачи из списка

## 16. Как работает AI в реальном времени
Я бы делал так:

**Сценарий**
Пользователь пишет:
> "добавь электрику и сантехнику после монолита для всех этажей"

**Шаг 1**
Frontend отправляет сообщение в AI chat endpoint

**Шаг 2**
MCP/LLM читает контекст проекта через tools

**Шаг 3**
LLM решает:
-   какие задачи создать
-   где
-   какие зависимости
-   нужен ли repeat by floor

**Шаг 4**
LLM **не пишет в БД сам**
Он вызывает tool-команды:
-   `create template`
-   `apply to zones`
-   `create dependencies`

**Шаг 5**
App API выполняет команды, пишет в Postgres

**Шаг 6**
Realtime gateway шлёт всем обновления

**Шаг 7**
Frontend видит новые задачи прямо в графике

## 17. Очень важное правило для AI
LLM **не должен иметь право делать “произвольный SQL”.**

**Только:**
-   читать через безопасные **query tools**
-   писать через валидированные **command tools**

То есть **AI работает как пользователь через API**, а не как root в базе.
