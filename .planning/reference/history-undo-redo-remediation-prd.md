# PRD: Remediation для истории действий Phase 44

## 1. Зачем нужен remediation

Phase 44 реализовал надёжную, но избыточную для v1 модель истории:

- `undo` создаёт новую `MutationGroup(origin='undo')`
- `redo` создаёт новую `MutationGroup(origin='redo')`
- пользователь видит технические последствия rollback/replay как обычные записи ленты
- UI перегружен статусами, redo-состояниями и внутренней логикой append-only журнала

Технически это консистентно, но продуктово даёт лишнюю сложность:

- история выглядит как журнал внутренних операций, а не как понятная лента действий
- `redo` как отдельный пользовательский сценарий усложняет mental model
- запись об откате появляется как новый элемент истории и засоряет список

Этот remediation PRD упрощает **пользовательскую модель**, не ломая уже построенный backend-фундамент.

## 2. Главная цель remediation

Пользователь должен видеть **простую линейную историю действий**, близкую по ощущению к истории в Google Sheets:

- есть список шагов
- есть текущая активная точка
- на любом предыдущем шаге можно нажать `Откатиться сюда`
- после отката более поздняя ветка считается сброшенной
- в ленте не появляются отдельные пользовательские записи “Undo” / “Redo”

Главный принцип remediation:

**Внутренний append-only log можно сохранить, но пользователь не должен видеть его как модель продукта.**

## 3. Что сохраняем из уже сделанного

Эти части текущей реализации считаются правильным фундаментом и сохраняются:

- `Project.version`
- authoritative commit path через `CommandService.commitCommand`
- `MutationGroup` как единица user-visible действия
- `ProjectEvent.groupId`
- `ordinal`
- `inverseCommand`
- `requestContextId`
- правило `one agent turn = one MutationGroup`
- линейный versioned pipeline и optimistic concurrency

То есть remediation не отменяет основу истории. Он меняет **контракт продукта поверх неё**.

## 4. Что считаем ошибкой v1

Для v1 ошибкой считаются следующие продуктовые решения:

- показывать `undo`-группы как обычные записи истории
- показывать `redo`-группы как обычные записи истории
- делать `redo` центральной частью пользовательского UX
- строить UI вокруг `undoable` / `redoable` на каждой карточке
- отображать историю как отражение внутреннего event log

Это допустимо как внутренний механизм, но не как пользовательская модель.

## 5. Новая пользовательская модель

### 5.1 Линейная активная история

История в UI — это **линейный список активных действий**.

Каждая запись показывает:

- время
- автора
- понятный заголовок действия
- при необходимости краткий secondary text
- одну явную кнопку: `Откатиться сюда`

Пользователь не должен видеть:

- отдельные записи `Undo — ...`
- отдельные записи `Redo — ...`
- typed divergence-механику
- внутренние связи `undoneByGroupId` / `redoOfGroupId`

### 5.2 Откат к точке

Вместо сценария “undo/redo как новые действия” пользователь работает с одним действием:

- `Откатиться сюда`

Поведение:

1. Пользователь выбирает запись истории
2. Сервер откатывает текущий хвост до выбранной точки
3. После этого выбранная запись становится текущей активной точкой
4. Более поздняя ветка считается сброшенной для пользовательской истории

### 5.3 После отката

После отката:

- в пользовательской ленте не появляется новый шаг `Undo`
- в пользовательской ленте не появляется новый шаг `Redo`
- пользователь видит результат как смену текущей активной точки истории
- дальнейшие изменения создают новую активную линию поверх выбранной точки

## 6. Что не меняем в схеме данных

Для remediation **не требуется новая миграция**.

Используем уже существующие поля:

- `MutationGroup.id`
- `MutationGroup.baseVersion`
- `MutationGroup.newVersion`
- `MutationGroup.origin`
- `MutationGroup.status`
- `MutationGroup.undoneByGroupId`
- `MutationGroup.redoOfGroupId`
- `ProjectEvent.groupId`
- `ProjectEvent.ordinal`
- `ProjectEvent.inverseCommand`

То есть remediation делается на уровне:

- domain logic
- history service
- API contract
- UI

## 7. Новый backend-контракт

### 7.1 Вместо публичного undo/redo

Публичная продуктовая операция должна быть одна:

#### `POST /api/history/:groupId/revert`

Смысл:

- привести проект к состоянию после выбранной группы
- откатить только текущий хвост активной истории

### 7.2 Как выполняется откат

Сервер **не** должен replay-ить всю историю с нуля.

Сервер должен:

1. найти текущую активную линейную историю
2. взять только группы после `targetGroupId`
3. для этих групп взять `inverseCommand`
4. применить inverse-команды в обратном порядке
5. выполнить откат через существующий authoritative command path

То есть rollback идёт **по хвосту**, а не по всей истории проекта.

### 7.3 Что остаётся внутренним

Если для надёжности backend продолжает создавать технические rollback-группы, это допустимо.

Но они:

- не должны отображаться пользователю как элементы истории
- не должны формировать основной UX
- не должны требовать от UI объяснять `redo`, divergence и replay semantics

## 8. Новый history API

### 8.1 `GET /api/history`

API должен возвращать только **пользовательскую активную линейную историю**.

Не должен возвращать как обычные действия:

- `origin='undo'`
- `origin='redo'`

Или, если возвращает их внутренне, то UI-contract обязан их отфильтровать/скрыть до уровня продукта.

### 8.2 Запись истории

Минимальный контракт записи:

```ts
{
  id: string;
  actorType: 'user' | 'agent' | 'system';
  title: string;
  createdAt: string;
  isCurrent?: boolean;
  canRevert?: boolean;
}
```

Допустимы дополнительные поля, но UI не должен строиться вокруг `redoable`.

## 9. Новый UI-контракт

### 9.1 Карточка истории

Карточка должна быть компактной и понятной:

- иконка автора
- короткий title
- время
- кнопка `Откатиться сюда`

Не нужно в v1:

- отдельная кнопка `Повторить`
- отдельный чип `Применено`
- технические названия команд
- отдельные записи “Undo — ...”

### 9.2 Текущая точка

Нужно явно показывать текущую активную точку истории.

Например:

- выделение строки
- метка `Текущая`
- отсутствие кнопки `Откатиться сюда` у текущего шага

### 9.3 Горячие клавиши

После remediation допускаются два варианта:

1. временно оставить только `Ctrl+Z` как откат на один шаг назад
2. полностью перевести клавиатурный сценарий на линейную модель без публичного `redo`

Предпочтительный вариант v1.1:

- `Ctrl+Z` = откат на предыдущую активную запись
- без пользовательского `Ctrl+Shift+Z`

## 10. Что явно убираем из продуктовой модели

После remediation продуктовая модель **не должна опираться** на:

- `redo` как пользовательскую кнопку на карточке
- `redoable` как основной визуальный статус
- typed divergence reason как часть UI
- новую видимую запись истории при rollback

Это всё может остаться глубоко внутри реализации, но не в интерфейсе пользователя.

## 11. Ограничения remediation

Этот remediation:

- не требует snapshot-first архитектуры
- не требует materialized snapshots
- не требует новой миграции
- не требует full replay всей истории

Этот remediation **не** пытается:

- ввести git-like ветки
- делать time travel preview
- восстанавливать произвольные альтернативные ветки
- строить diff UI

## 12. Критерии успеха

Remediation успешен, если одновременно выполняются все условия:

1. История в UI выглядит как простой линейный список действий, а не как технический журнал rollback/replay.
2. Пользователь может выбрать прошлую запись и нажать `Откатиться сюда`.
3. После отката в истории не появляется новый видимый пункт `Undo`.
4. Более поздние шаги не создают путаницу в ленте и считаются сброшенной веткой.
5. Backend не replay-ит всю историю с нуля, а откатывает только хвост.
6. Для первого remediation-шага не потребовались новые миграции.

## 13. План внедрения

### Step 1. Скрыть лишнюю сложность из UI

- убрать пользовательский `redo`
- убрать отображение `undo` / `redo` групп как отдельных записей
- заменить per-row `undo/redo` на `Откатиться сюда`

### Step 2. Упростить history API

- отдать только активную линейную историю
- перестать строить UI вокруг `redoable`

### Step 3. Упростить server behavior

- ввести сценарий `revertTo(groupId)`
- откатывать только хвост активной истории
- сохранить текущий command pipeline

### Step 4. При необходимости later cleanup

Если после стабилизации останутся реально неиспользуемые поля или ветки логики, можно сделать отдельную cleanup-фазу.

Но это не prerequisite для remediation.

## 14. Ключевое решение remediation

**Текущая модель данных не является ошибкой. Ошибка — показывать внутреннюю undo/redo-механику как пользовательскую модель продукта.**

Правильное упрощение:

- оставить надёжный backend-фундамент
- убрать тяжёлую пользовательскую модель
- дать пользователю простую линейную историю с действием `Откатиться сюда`
