давай вообще проанализируем что можно добавить в MCP ещё? Меня смущает что ты сказал что у нас regex на фразу? ЧЕГО? а что, агент не сам распознаёт что вызывать?


• Да, агент сам решает, какой MCP tool вызывать. regex не управлял выбором tool вообще.

  Он был только в сервере, в agent.ts, для служебного вопроса:

  - считать ли этот пользовательский запрос мутацией;
  - включать ли защитный режим;
  - проверять ли, что после ответа была реальная мутация в БД;
  - запускать ли retry, если модель соврала и не вызвала tool.

  То есть схема такая:

  1. LLM читает запрос и сама выбирает get_tasks, create_task, update_task и т.д.
  2. Сервер параллельно пытается понять: “это вообще запрос на изменение или просто вопрос?”
  3. Если это изменение, сервер после ответа проверяет факт мутации.

  Проблема была в шаге 2: русский запрос иногда ошибочно считался не-мутирующим, и сервер не включал верификацию.

  Что стоит добавить в MCP
  Сейчас у вас MCP слишком низкоуровневый для сложных фраз. Поэтому модель должна сама раскладывать команду на серию операций, и именно тут она иногда “рассуждает правильно, но
  tool call не делает”. Я бы усилил MCP такими инструментами:

  1. insert_task_after

  - Вставить одну задачу после указанной задачи.
  - MCP сам считает даты, зависимости и место в цепочке.

  2. insert_task_block_after

  - Вставить блок задач после указанной задачи.
  - На вход:
      - anchorTaskId
      - список задач с длительностями
      - тип связи внутри блока
      - опционально связь блока с последующей задачей
  - Это как раз ваш кейс “Подготовка ТЗ (сбор, анализ, дип ресёрч)”.

  3. append_task_to_end

  - Добавить задачу в конец текущего плана.
  - Без необходимости модели самой искать последнюю задачу и руками считать дату.

  4. split_task

  - Разбить одну задачу на N частей или по массиву длительностей.
  - У вас это уже частый сценарий.

  5. move_task_after

  - Переставить существующую задачу после другой с перерасчётом дат и зависимостей.

  6. rename_task_by_name или find_tasks

  - Сейчас модель часто ищет задачу по имени через get_tasks и потом сама сопоставляет.
  - Нужен явный инструмент поиска:
      - точное совпадение
      - частичное совпадение
      - список кандидатов при неоднозначности

  7. plan_snapshot или get_schedule_context

  - Отдавать не просто список задач, а:
      - последнюю задачу
      - критическую последовательность
      - порядок задач
      - ближайшие свободные окна
  - Это сильно уменьшит “рассуждение в голове модели”.

  Что я бы поменял не regex-ом, а архитектурно
  1. Считать mutation по факту tool calls


  2. Для ambiguous запросов делать двухфазный режим

  - первая фаза: агент обязан только понять intent и выбрать action type;
  - вторая фаза: исполнение через tools.
  - Но это сложнее и медленнее.

  3. Ввести tool-level policy

  - если финальный ответ содержит слова вроде “добавлено/изменено/удалено”, а mutation tools не вызывались, ответ автоматически invalid.
  - Это уже частично у нас есть, но стоит сделать основным механизмом, а не fallback.

  Что я рекомендую сделать дальше
  Самый практичный пакет:

  1. Добавить в MCP:

  - find_tasks
  - insert_task_block_after
  - split_task

  2. На сервере упростить логику:

  - не пытаться слишком умно угадывать mutation по тексту;
  - ориентироваться прежде всего на факт tool calls и на изменение snapshot.

  3. Для сложных команд типа “после X добавь блок работ ...” предпочитать один высокоуровневый MCP tool, а не серию из 3-5 низкоуровневых вызовов.
