ЧАСТЬ 1: ОПТИМИЗАЦИЯ СТОИМОСТИ API
Текущая стоимость: ~15 ₽ за один график. Цель: снизить до 3–5 ₽ без потери качества.

1.1 Почему сейчас дорого
Типичная причина высокой стоимости — неоптимальная работа с контекстом: каждый раз передаётся весь системный промпт, история диалога и данные о задачах заново. При большом числе задач токены накапливаются очень быстро.

Источник токенов	Примерный вес	Оптимизируемо?
Системный промпт (описание инструментов)	~2 000 токенов	Да — кэш
История диалога (повтор каждый turn)	~1 000–5 000 токенов	Да — кэш
Данные сметы / задач в промпте	~3 000–8 000 токенов	Да — пагинация
Ответ модели (output)	~500–2 000 токенов	Частично
Tool results (ответы MCP)	~500–1 500 токенов	Да — сжатие

1.2 Qwen SDK: сессии и multi-turn
Главная возможность Qwen Code SDK для экономии — поддержка multi-turn conversation через AsyncIterable. Вместо того чтобы каждый раз стартовать новую сессию с нуля, контекст сохраняется внутри одной сессии.

Ключевая возможность из документации
query() принимает AsyncIterable<SDKUserMessage> для multi-turn разговора. Сессия живёт пока открыт итератор. Метод q.getSessionId() позволяет идентифицировать сессию и переподключаться.

Как использовать multi-turn для экономии:
При первом запросе — создаёшь сессию, передаёшь системный промпт и данные сметы один раз. Все последующие сообщения пользователя в рамках работы с этим проектом идут в ту же сессию. Контекст уже в памяти модели — не нужно передавать повторно.

Пример реализации на Qwen SDK:
// Создаём генератор сообщений для одной рабочей сессии
async function* projectSession(projectId, userMessages) {
  // Первое сообщение — передаём контекст проекта один раз
  yield { type: 'user', session_id: projectId,
    message: { role: 'user', content: buildProjectContext(projectId) } };
  // Дальше просто принимаем команды пользователя
  for await (const msg of userMessages) {
    yield { type: 'user', session_id: projectId, message: msg };
  }
}
// Сессия живёт пока пользователь работает с проектом
const session = query({ prompt: projectSession(id, stream), options: { permissionMode: 'yolo', mcpServers: { gantt: myMcpServer } } });

1.3 Все стратегии экономии

Стратегия 1 — Prompt Caching (самая мощная)
Системный промпт с описанием инструментов MCP не меняется между запросами. Если использовать Anthropic API напрямую (Claude) — включить cache_control на системный промпт. Для Qwen — аналогично через параметр кэширования в настройках модели. Экономия: 50–70% стоимости input токенов.

Стратегия 2 — Не гонять задачи в промпте
Вместо того чтобы передавать агенту весь список задач проекта при каждом запросе — пусть агент сам запрашивает нужные задачи через MCP инструмент get_task(id). Агент получает только то что нужно для конкретного действия.

Стратегия 3 — Сжатие tool results
MCP инструменты возвращают только нужные поля, без лишних метаданных. Например get_all_tasks возвращает [id, title, start_date, duration] вместо полного объекта. Экономия 30–40% на tool result токенах.

Стратегия 4 — Умный выбор модели
Задача	Рекомендуемая модель	Стоимость	Обоснование
Парсинг структурированной сметы	qwen-turbo / claude-haiku	Дёшево	Структурная задача, не нужен большой ум
Семантическое понимание нестандартной сметы	qwen-max / claude-sonnet	Дороже	Нужно понимание контекста
Простые команды (сдвинь, удали)	qwen-plus	Средне	Простой tool use
Анализ рисков / критическая цепь	qwen-max / claude-sonnet	Дороже	Сложный анализ

Стратегия 5 — Батчинг при создании задач
Вместо того чтобы агент вызывал create_task 100 раз подряд — добавить инструмент create_tasks_batch(tasks[]) который принимает массив и сохраняет всё за один вызов. Сокращает число round-trip с моделью в 5–10 раз.

Стратегия 6 — Ограничение сессии через maxSessionTurns
Qwen SDK поддерживает maxSessionTurns. Для задачи парсинга сметы — ограничить 20 turns. Для чата с пользователем — 50 turns. Если сессия исчерпана — создать новую с сжатым контекстом (summary предыдущей работы).
