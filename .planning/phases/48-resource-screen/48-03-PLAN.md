Existing endpoints:
- `GET /api/resources?projectId=...` returns `{ resources }` and includes inactive resources.
- `POST /api/resources` accepts `{ name, type, scope, projectId? }`.
- `PATCH /api/resources/:resourceId` accepts `name`, `type`, `scope`, and `isActive`; existing route sends the service response.

Constraints:
- Before coding against PATCH response, verify its shape in tests or normalize defensively with existing `normalizeProjectResource`.
- Do not implement a new backend endpoint.
- Deactivate/activate must use textual confirmation copy from UI-SPEC.
- Readonly/no accessToken state disables catalog mutation actions and shows explanatory copy.
</context>
