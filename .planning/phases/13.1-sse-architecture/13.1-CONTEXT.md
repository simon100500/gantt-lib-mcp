# Phase 13.1: SSE Architecture - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Полная замена WebSocket на Server-Sent Events (SSE) с миграцией на PostgreSQL.

**Что входит:**
- Удалить WebSocket (`/ws`), заменить на SSE endpoints
- PostgreSQL в отдельном контейнере CapRover (пользователь разворачивает самостоятельно)
- Prisma ORM для работы с PostgreSQL
- PostgreSQL триггеры с NOTIFY/LISTEN для real-time updates
- Два SSE endpoint: `/stream/tasks` (обновления Ганта) и `/stream/ai` (стриминг агента)

**Не входит:**
- Миграция данных из SQLite (пользователи вводят данные заново)
- WebSocket (полностью убирается)
- Новые фичи приложения

</domain>

<decisions>
## Implementation Decisions

### Архитектура: SSE вместо WebSocket
- **Полная замена:** убрать `/ws`, `packages/server/src/ws.ts`
- **Два SSE endpoints:**
  - `GET /stream/tasks` — обновления Ганта (задачи созданы/изменены/удалены)
  - `GET /stream/ai` — стриминг ответа агента (токены по мере генерации)
- **SSE формат:** стандартный `text/event-stream` с `data:` prefixed JSON
- **Клиент:** `EventSource` браузера для `/stream/tasks`, `fetch` с stream reader для `/stream/ai`

### PostgreSQL: отдельный контейнер
- **Развёртывание:** пользователь самостоятельно разворачивает PostgreSQL в CapRover
- **Credential'ы:** передаются через env vars (`DATABASE_URL`)
- **Без миграции данных:** существующие данные из SQLite не переносятся
- **ORM:** Prisma (вместо @libsql/client)

### Real-time: PostgreSQL NOTIFY/LISTEN
- **Триггеры в БД:** создаются через миграцию Prisma
- **Таблицы с триггерами:** `tasks`, `dependencies` (любой INSERT/UPDATE/DELETE)
- **NOTIFY payload:** JSON `{ action, project_id, task_id, task }`
- **Listener:** отдельный `pg.Client` (не пул) с `LISTEN tasks_channel`
- **Роутинг:** по `project_id` из payload находим нужные SSE соединения

### SSE connection registry
- **Структура:** `Map<projectId, Set<reply.raw>>` (как было с WebSocket)
- **Аутентификация:** через JWT в `Authorization` header при подключении
- **Heartbeat:** периодический `:heartbeat\n\n` для detecting dead connections
- **Cleanup:** при `close` событии удаляем из Map

### Agent streaming через SSE
- **Endpoint:** `GET /stream/ai`
- **Auth:** JWT в Authorization header
- **Streaming:** накапливаем токены из qwen-code SDK, пишем `data: { content: "..." }`
- **Final message:** `data: { type: "done" }`

### Nginx конфигурация
- **HTTP/2:** включить для мультиплексирования (убирает лимит 6 соединений)
- **proxy_buffering off:** критично для SSE
- **proxy_cache off:** не кэшировать stream

### Dependencies
- **Добавить:** `@prisma/client`, `prisma`, `pg` (для LISTEN client)
- **Убрать:** `@fastify/websocket`, `@libsql/client`

### Claude's Discretion
- Интервал heartbeat (30 сек или другой)
- Формат SSE payload (может отличаться для tasks vs ai)
- Обработка ошибок LISTEN (reconnect логика)
- Нужно ли сжатие для SSE

</decisions>

<specifics>
## Specific Ideas

**SSE payload формат для /stream/tasks:**
```
data: {"action":"insert","project_id":"abc","task":{"id":"123","name":"Task",...}}

data: {"action":"update","project_id":"abc","task_id":"123","changes":{...}}

data: {"action":"delete","project_id":"abc","task_id":"123"}
```

**SSE payload формат для /stream/ai:**
```
data: {"content":"Добавлена"}

data: {"content":" задача"}

data: {"type":"done"}
```

**PostgreSQL триггер (из обсуждения):**
```sql
CREATE OR REPLACE FUNCTION notify_task_change()
RETURNS trigger AS $$
BEGIN
  PERFORM pg_notify(
    'tasks_channel',
    json_build_object(
      'action', TG_OP,
      'project_id', NEW.project_id,
      'task_id', NEW.id,
      'task', row_to_json(NEW)
    )::text
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

**CapRover setup:**
- PostgreSQL app: пользователь создаёт отдельно
- DATABASE_URL: `postgresql://user:pass@pg-app:5432/gantt`
- В main app: DATABASE_URL env var указывает на pg-app

</specifics>
