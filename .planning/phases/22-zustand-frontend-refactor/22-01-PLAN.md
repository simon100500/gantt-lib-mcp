<objective>
Install Zustand in the web package and establish the first two frontend stores: chat and UI. This plan also removes the module-level saving-state listener pattern from `useBatchTaskUpdate` and redirects save-state writes into the new UI store.
</objective>

<tasks>
<task type="auto">
  <name>Task 1: Add Zustand and create chat/UI stores</name>
  <files>packages/web/package.json, package-lock.json, packages/web/src/stores/useChatStore.ts, packages/web/src/stores/useUIStore.ts</files>
  <action>
  Add `zustand` to `packages/web` and create the first store layer in `src/stores/`.

  `useChatStore.ts` must own:
  - chat messages
  - streaming text
  - AI thinking state
  - `addMessage`, `appendToken`, `finishStreaming`, `setError`, and `reset`

  `useUIStore.ts` must own:
  - workspace mode
  - modal/sidebar toggles
  - view mode
  - task list visibility
  - auto-schedule toggle
  - expired-task highlighting
  - validation errors
  - share status
  - saving state

  Keep the store types local to the store modules unless a shared type is clearly reused elsewhere. `addMessage` should use `crypto.randomUUID()` instead of a module counter.
  </action>
  <verify>
    <automated>rg -n "zustand" packages/web/package.json</automated>
    <automated>rg -n "crypto\\.randomUUID|appendToken|finishStreaming|setError|reset" packages/web/src/stores/useChatStore.ts</automated>
    <automated>rg -n "setSavingState|workspace|shareStatus|validationErrors" packages/web/src/stores/useUIStore.ts</automated>
  </verify>
  <done>
    Zustand is available in the web package, and the chat/UI state model exists as store-backed modules.
  </done>
</task>

<task type="auto">
  <name>Task 2: Move saving-state plumbing out of useBatchTaskUpdate</name>
  <files>packages/web/src/hooks/useBatchTaskUpdate.ts, packages/web/src/stores/useUIStore.ts</files>
  <action>
  Remove the custom module-level saving-state listener set from `useBatchTaskUpdate.ts`.

  Replace the internal `globalSavingState`/listener pattern with direct UI store writes:
  - write save-state transitions through `useUIStore.getState().setSavingState(...)`
  - read the current save state from the UI store selector instead of a local subscription list
  - keep the hook behavior and returned `savingState` stable for the rest of the migration

  The hook should remain a hook; only the save-state transport changes.
  </action>
  <verify>
    <automated>rg -n "globalSavingState|listeners|add\\(|delete\\(" packages/web/src/hooks/useBatchTaskUpdate.ts</automated>
    <automated>rg -n "useUIStore\\.getState\\(\\)\\.setSavingState|savingState" packages/web/src/hooks/useBatchTaskUpdate.ts</automated>
  </verify>
  <done>
    Save-state updates flow through Zustand, and `useBatchTaskUpdate` no longer maintains its own listener registry.
  </done>
</task>
</tasks>
